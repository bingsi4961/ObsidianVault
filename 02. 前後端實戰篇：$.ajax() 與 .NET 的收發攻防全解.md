# 02. 前後端實戰篇：$.ajax() 與 .NET 的收發攻防全解

> **閱讀順序**：本篇是三部曲的第二篇。請先讀完《01. 觀念基礎篇》，讀完本篇後再看《03. $.ajax() 完整範本》。

上一篇我們把 HTTP 傳輸的地基打好了。這一篇要進入你每天上班都會碰到的實戰：**jQuery 怎麼把資料寄出去、.NET 怎麼把資料拆出來、後端怎麼回傳、出錯了怎麼抓蟲**。

先建立一個貫穿全篇的核心比喻，後面所有的「坑」都靠它解釋：

> 💡 **jQuery 其實只是個「秘書」，真正的「郵差」是瀏覽器 🌐。** jQuery 底層是呼叫瀏覽器內建的傳輸功能（`XMLHttpRequest` 或 `fetch` API）來送信的。秘書負責打包、貼標籤，郵差負責運送。很多災難都是因為「秘書自作聰明」或「秘書和郵差搶工作」造成的。

## 1. jQuery 發送資料（Request）

### 1.1 預設行為：平信信封

先回答一個基本問題：**$.ajax() 的 `type` 預設值是什麼？**

答案是 **GET** ✉️。如果你沒有特別寫 `type: 'POST'`，jQuery 就會預設派郵差送出「明信片（GET）」。這也是為什麼很多開發者做單純查詢時 AJAX 寫得特別短，因為預設的 GET 剛好符合需求。

再來，**jQuery 預設的 Content-Type 是什麼？**

跟傳統表單一樣，是 `application/x-www-form-urlencoded`（平信信封）：

```javascript
// 最簡單的 AJAX 請求
$.ajax({
    url: '/api/saveUser',
    type: 'POST',
    data: {
        username: 'john',
        email: 'john@asus.com',
        age: 30
    },
    success: function(response) {
        console.log('儲存成功');
    }
});
```

實際發送的 HTTP Request：

```
POST /api/saveUser HTTP/1.1
Content-Type: application/x-www-form-urlencoded; charset=UTF-8

username=john&email=john%40asus.com&age=30
```

**背後的脈絡（歷史包袱）**：jQuery 誕生的年代，JSON 還不是主流，當時的絕對霸主就是平信。所以 jQuery 的底層被設計成一個「盡責的平信翻譯員」——只要你塞一個 JavaScript 物件給 `data`，它就會啟動內部的 `$.param()` 函數（這個機制的開關叫 `processData`），強制把物件拆解成平信格式。

例如 `data: { name: "john", age: 18 }` 會被壓扁成純文字：

```
name=john&age=18
```

> 📌 **小結**：無論是 HTML 網頁表單還是 jQuery AJAX，只要你沒特別指定，系統預設的寄信方式都是「平信信封 (x-www-form-urlencoded)」。

### 1.2 發送 JSON：contentType + JSON.stringify 兩步到位

如果要傳的資料很複雜（例如包含陣列、多層子物件），硬要壓扁成 `a=1&b=2` 會非常痛苦，後端也很難拆解。這時候就改用 JSON，但**必須兩步到位**：

```javascript
// 發送 JSON 格式
$.ajax({
    url: '/api/saveUser',
    type: 'POST',
    contentType: 'application/json',  // (1) 貼標籤：告訴伺服器我送的是 JSON
    data: JSON.stringify({            // (2) 壓扁內容：必須將物件轉成 JSON 字串
        username: 'john',
        email: 'john@asus.com',
        age: 30,
        hobbies: ['reading', 'coding']
    }),
    success: function(response) {
        console.log('儲存成功');
    }
});
```

實際發送的完整 HTTP 請求會長這樣：

```
POST /api/saveUser HTTP/1.1
Host: localhost:5000
Content-Type: application/json; charset=utf-8
Content-Length: 84

{"username":"john","email":"john@asus.com","age":30,"hobbies":["reading","coding"]}
```

**觀念釐清：標籤與內容是兩件事** 🏷️📦

`contentType` 只是包裹外面的「分類標籤」，而你怎麼組合資料則是「實際裝進包裹裡的內容」。**這兩者必須一致**。如果箱子外貼「JSON 格式」，裡面卻塞平信格式的資料，伺服器用拆 JSON 的方式去拆平信，程式就會直接當機報錯。

**為什麼需要 JSON.stringify？**

在 JavaScript 裡，`{ name: "john", age: 30 }` 是一個活生生、有結構的「立體物件 (Object)」。但 HTTP 網路傳輸只能傳送「純文字」。就像你做了一個樂高城堡，沒辦法直接塞進信封袋寄出——必須先拆解、畫成平面說明書（字串）才能寄。`JSON.stringify()` 的功能，就是把「立體物件」壓扁成「平面的純文字字串」。

### 1.3 魔王級地雷：忘記 JSON.stringify() 會發生什麼事？

```javascript
// ❌ 錯誤：說是 JSON 但傳送物件
$.ajax({
    contentType: 'application/json',
    data: { name: 'john' }           // 這裡是物件，不是 JSON 字串
});

// ✅ 正確：轉換成 JSON 字串
$.ajax({
    contentType: 'application/json',
    data: JSON.stringify({ name: 'john' })
});
```

**jQuery 不會聰明地自動幫你 stringify**。`contentType` 只是給伺服器看的標籤，**不會改變 jQuery 處理資料的方式**。災難流程如下：

1. **jQuery 的處理**：不管你標籤貼什麼，它照老規矩把物件拆成平信格式 `name=john`。
2. **伺服器的視角**：.NET 後端看到標籤寫著 JSON，滿心期待地拿「解析 JSON 的工具」去拆包裹，結果一打開，裡面的字串居然是 `name=john`（正確的 JSON 應該是 `{"name":"john"}`）。
3. **結果**：格式不符，直接報錯（通常是 400 Bad Request）。

更慘的是，如果你的物件很複雜，像 `{ user: { name: "john" } }`，jQuery 無法把「子物件」完美轉成平信，就會直接變成 `user=[object Object]` 這種無意義的亂碼。

💡 **生活比喻**：你在包裹外貼了「內附英文說明書（JSON 標籤）」，但忘記自己翻譯，直接把「中文說明書（立體物件）」丟給 jQuery。jQuery 按照舊習慣把它翻譯成「法文（平信格式）」。包裹寄到美國（伺服器），對方找英文翻譯員來讀，結果完全看不懂。

### 1.4 檔案上傳：FormData + 兩個 false

**為什麼檔案不能用 JSON？**

```javascript
// ❌ 這樣做是不可能的
var fileData = $('#fileInput')[0].files[0];
$.ajax({
    contentType: 'application/json',
    data: JSON.stringify({
        username: 'john',
        file: fileData  // 檔案是二進位資料，無法放入 JSON
    })
});
```

原因：

1. JSON 只能包含**文字資料**（字串、數字、布林值、陣列、物件）
2. 檔案是**二進位（Binary）資料**，無法用 JSON 表示。要把真實的蘋果（檔案）塞進傳真機（JSON）裡傳送是不可能的

**正確做法：使用 FormData（便當盒）**

```javascript
// 取得檔案
var selectedFile = $('#fileInput')[0].files[0];

// 建立 FormData（準備一個虛擬的便當盒）
var formData = new FormData();
formData.append('username', 'john');       // 放文字
formData.append('uploadFile', selectedFile); // 放檔案

// 發送檔案
$.ajax({
    url: '/api/uploadFile',
    type: 'POST',
    data: formData,
    processData: false,  // 魔法 1：叫 jQuery 閉嘴，不要把便當盒拆成平信 (a=1&b=2)
    contentType: false,  // 魔法 2：叫 jQuery 不要亂貼標籤，讓瀏覽器自己貼上帶有 boundary 的標籤
    success: function(response) {
        console.log('上傳成功');
    }
});
```

### 1.5 💣 超級大坑：千萬不要手寫 contentType: 'multipart/form-data'

```javascript
// ❌ 這是災難的開始
$.ajax({
    contentType: 'multipart/form-data',
    data: formData
});
```

這樣寫，伺服器會**完全收不到檔案**！

**為什麼？** 還記得便當盒的隔板（Boundary）嗎？為了讓伺服器知道飯、菜、肉的分界點在哪裡，瀏覽器在打包 multipart/form-data 時，會自動產生一串亂碼作為隔板。真正的標籤長這樣：

```
Content-Type: multipart/form-data; boundary=----WebKitFormBoundary7MA4YW
```

如果你手動寫死 `contentType: 'multipart/form-data'`，瀏覽器就會說：「喔，你自己設定了啊？那我就不幫你加 boundary 隔板囉！」結果後端看到標籤寫著「這是分隔便當盒」，滿心期待地準備拆解，卻在標籤上找不到「隔板長什麼樣子」的說明——它根本不知道怎麼把混在一起的文字和圖片分開，所有 FormData 都擠成一塊，只能兩手一攤直接報錯（通常回傳 400 Bad Request 或抓不到檔案）。

🚨 因為 boundary 那串亂碼（如 `7MA4YWxkTrZu0gW`）是**瀏覽器產生的，我們自己沒辦法處理**，所以只能設定 `contentType: false` 讓瀏覽器接手。

### 1.6 底層解惑：processData 和 contentType: false 到底在做什麼？

這一段是全篇最燒腦、也最值錢的底層邏輯，弄懂之後你就跟「通靈除錯」說再見了。

**Q1：processData: false 跟 JSON.stringify 是一樣的意思嗎？**

它們的「目的」都是為了防止 jQuery 破壞資料，但「作法」完全不同：

- **JSON.stringify（自己先動手）**：你把立體的樂高城堡，自己先畫成一張平面設計圖（純文字字串）。jQuery 一看：「喔，這已經是純文字了，我不需要把它拆成平信。」於是原封不動寄出。
- **processData: false（貼上「請勿動手」標籤）**：用在便當盒（FormData）上。便當盒是沒辦法壓成紙張的！你只能在盒外貼一張「請勿擠壓」貼紙，警告 jQuery：「這是特殊的立體包裹，不要試圖把它壓扁成 a=1&b=2，請原封不動交給郵差。」

**Q2：JSON 的情境為什麼「不用」寫 processData: false？這不是矛盾嗎？**

秘密藏在 jQuery 對 `processData` 預設值的判斷邏輯裡。`processData` 預設為 `true`，它的真實運作規則是：

> 「如果收到的是『物件 (Object)』，我就轉成平信；但如果收到的已經是『字串 (String)』，我就什麼都不做，直接放行。」

|傳送情境|傳入 data 的型態|processData 狀態|jQuery 的實際動作|
|---|---|---|---|
|❌ 錯誤的 JSON<br>`data: { name: "john" }`|物件|預設（啟動轉換）|變成平信 `name=john`，後端解析 JSON 失敗|
|✅ 正確的 JSON<br>`data: JSON.stringify(...)`|**字串**|預設（啟動轉換）|發現已經是字串，直接放行不轉換|
|✅ 檔案便當盒<br>`data: new FormData()`|物件|**強制設為 false**|FormData 是物件，不設 false 的話 jQuery 會試圖把它轉成字串然後出錯；設 false 強制放行|

📌 JSON 之所以不用寫 `processData: false`，是因為 `JSON.stringify` 已經把它變成了字串，成功「騙」過了 jQuery 的平信轉換機制！

**Q3：contentType: false 之後，瀏覽器怎麼知道要貼什麼標籤？它怎麼認得 FormData？**

流程是這樣的：

1. jQuery 看到你設定了 `contentType: false`，秘書就說：「好，我不在包裹外箱寫任何標籤，保持空白。」
2. 秘書把包裹轉交給郵差（瀏覽器）。
3. **瀏覽器的智慧介入**：郵差接過包裹一看：「咦！這是一個 FormData 格式的便當盒啊！」於是主動幫你貼上 `Content-Type: multipart/form-data; boundary=----WebKit...`。

那瀏覽器是怎麼「認得」FormData 的？它**不是**單純判斷「物件還是字串」——`FormData` 不是我們隨便寫的普通物件，它是**瀏覽器內建的「官方特製包裹 API」**。

💡 如果普通的 JavaScript 物件 `{}` 是你自己找來的「一般紙箱」，那 `FormData` 就是郵局發行的「官方專用便利箱」。郵差在底層「認得」這個專屬型別，只要看到它，就會自動啟動處理檔案的特殊流程（自動蓋上 multipart 專用郵戳並加上 boundary）。

反過來說，如果你**沒有**設定 `contentType: false`，秘書（jQuery）就會自作聰明先貼上「平信標籤」；郵差看到已經有標籤了，就不會再幫你貼便當盒的專屬標籤，後端收到就解析失敗。

**Q4：回傳的時候，包裹是 jQuery 收，還是瀏覽器收？**

在網路上，包裹「物理上」**絕對是先寄到郵差（瀏覽器）手裡的**——瀏覽器的底層網路連線功能，確實接收到了伺服器回傳的位元組資料。但因為這個請求是透過 `$.ajax` 發出的「非同步請求（XMLHttpRequest）」，瀏覽器收到包裹後，不會把它丟給「畫面渲染引擎」去畫網頁，也不會觸發「下載視窗」，而是直接把資料塞進 JavaScript 的「記憶體」裡，然後呼叫你的 `success: function(data)`。

💡 就像你平常請郵差拿信，郵差會直接在大廳把信件內容展示給你讀（一般的網頁載入）。但今天你指派了秘書（jQuery AJAX）處理機密任務，秘書跟郵差說：「拿回來之後，請偷偷從後門直接交到我的辦公桌上，不要去大廳展示！」——這個特性稍後在「檔案下載」章節會引發一場災難，先記住它。

### 1.7 三種 AJAX 傳輸方式對比總結

|情境|資料包裝方式 (data)|contentType 設定|processData 設定|適用時機|
|---|---|---|---|---|
|純文字（平信）✉️|`data: { name: "john" }`|不寫（預設）|不寫（預設）|簡單的搜尋、表單送出|
|複雜結構（JSON）📖|`data: JSON.stringify(...)`|`'application/json'`|不寫（預設）|傳送陣列、多層級資料|
|包含檔案（便當盒）🍱|`data: new FormData()`|`false`|`false`|只要有上傳圖片、Excel 就必用|

> 📌 **記住：有檔案就用 FormData + contentType: false + processData: false。**

### 1.8 FormData 進階：同時上傳欄位值、陣列、JSON、檔案

FormData 這個便當盒的夾層很多，可以混裝各種資料：

```javascript
// ✅ 正確做法：使用 FormData
var formData = new FormData();

// 一般資料
formData.append('username', 'john');

// 陣列資料（同名 key 重複 append，.NET 才收得到）
formData.append('hobbies', 'reading');
formData.append('hobbies', 'coding');

// JSON 資料（轉成字串塞進夾層）
formData.append('preferences', JSON.stringify({
    theme: 'dark',
    language: 'zh-TW'
}));

// 檔案資料
formData.append('profilePhoto', $('#photoInput')[0].files[0]);

$.ajax({
    url: '/api/updateProfile',
    type: 'POST',
    data: formData,
    processData: false,
    contentType: false  // 🚨 讓瀏覽器自動貼 multipart/form-data
});
```

> ⚠️ 你原本的筆記中，陣列是寫成 `formData.append('hobbies[]', 'reading')`（名稱帶中括號）。這是 **PHP 風格**的命名，**.NET / .NET Core 的 Model Binding 並不認得 `hobbies[]` 這個名稱**，會導致後端的 `List<string> hobbies` 收到空集合——這正好跟後面 `traditional` 章節講的「.NET 看到 `ids[]` 多了括號就抓不到」是同一件事。串接 .NET 後端時，請改成**同名 key 重複 append**（如上），後端就能自動收集成 `List<string>`。

實際的 HTTP Body 長這樣（修正命名後）：

```
POST /api/updateProfile HTTP/1.1
Host: localhost:44301
Content-Type: multipart/form-data; boundary=----WebKitFormBoundary7MA4YWxkTrZu0gW
Content-Length: 1234

------WebKitFormBoundary7MA4YWxkTrZu0gW
Content-Disposition: form-data; name="username"

john
------WebKitFormBoundary7MA4YWxkTrZu0gW
Content-Disposition: form-data; name="hobbies"

reading
------WebKitFormBoundary7MA4YWxkTrZu0gW
Content-Disposition: form-data; name="hobbies"

coding
------WebKitFormBoundary7MA4YWxkTrZu0gW
Content-Disposition: form-data; name="preferences"

{"theme":"dark","language":"zh-TW"}
------WebKitFormBoundary7MA4YWxkTrZu0gW
Content-Disposition: form-data; name="profilePhoto"; filename="avatar.jpg"
Content-Type: image/jpeg

[二進位檔案內容 - 實際的圖片資料]

------WebKitFormBoundary7MA4YWxkTrZu0gW--
```

後端接收（.NET Core）：

```csharp
[HttpPost]
public IActionResult UpdateProfile(string username, List<string> hobbies, string preferences, IFormFile profilePhoto)
{
    try
    {
        // 1. username 直接可用
        Console.WriteLine($"使用者名稱: {username}");

        // 2. 處理 hobbies 陣列
        if (hobbies != null && hobbies.Count > 0)
        {
            Console.WriteLine($"興趣愛好: {string.Join(", ", hobbies)}");
        }

        // 3. preferences 是 JSON 字串，需要反序列化
        var preferencesObj = JsonSerializer.Deserialize<UserPreferences>(preferences);
        Console.WriteLine($"主題: {preferencesObj.Theme}, 語言: {preferencesObj.Language}");

        // 4. 處理上傳檔案
        if (profilePhoto != null && profilePhoto.Length > 0)
        {
            var fileName = Path.GetFileName(profilePhoto.FileName);
            var filePath = Path.Combine("uploads", fileName);

            using (var stream = new FileStream(filePath, FileMode.Create))
            {
                profilePhoto.CopyTo(stream);
            }

            Console.WriteLine($"檔案已上傳: {fileName}");
        }

        return Ok(new {
            success = true,
            message = "資料更新成功"
        });
    }
    catch (Exception ex)
    {
        Console.WriteLine($"錯誤: {ex.Message}");
        return BadRequest(new { success = false, message = ex.Message });
    }
}
```

### 1.9 traditional 設定：平信陣列的世紀之亂

這個設定非常經典，可以說是每一位串接 .NET 後端的前端工程師「必踩的坑」。

**實務上解決什麼問題？** 當你要用**平信格式**（未指定 contentType）傳送「一組陣列」給後端時（例如一口氣刪除 3 個員工，傳 `ids: [1, 2, 3]`），資料到了 .NET 後端，C# 的 `List<int> ids` 經常收到 `Count = 0` 的空陣列。`traditional` 這個開關，就是用來解決「前端陣列」與「後端 List」在平信格式下溝通不良的問題。

**背後的脈絡**：把物件壓扁成平信的過程叫「序列化（Serialization）」，`traditional` 就是 jQuery 序列化引擎裡的一個微調開關，專門控制「陣列該怎麼被壓扁成字串」。jQuery 1.4 版以前大家用傳統格式；後來為了配合 PHP 和 Ruby 的喜好，jQuery 改變了預設規則——偏偏我們用的 .NET (C#) 還是喜歡舊的那套傳統規則！

💡 **生活比喻**：幫同事買三杯紅茶——

- 現代新寫法（jQuery 預設）：訂單寫「飲料[第一杯]=紅茶 & 飲料[第二杯]=紅茶...」，雞婆地幫每個項目加上中括號 `[]`
- 傳統寫法（`traditional: true`）：直白地寫「飲料=紅茶 & 飲料=紅茶 & 飲料=紅茶」

```javascript
$.ajax({
    url: '/api/DeleteEmployees',
    type: 'POST',
    data: {
        department: 'IT',
        ids: [1, 2, 3]  // 這是一個陣列（注意：沒有轉 JSON）
    },
    traditional: true   // 🚨 串接 .NET 時必加！
});
```

|traditional 設定|被壓扁後的平信長相 (Payload)|.NET 的反應|
|---|---|---|
|`false`（預設值）|`department=IT&ids[]=1&ids[]=2&ids[]=3`|❌ **抓不到陣列！** .NET 看到 `ids[]` 多了括號，覺得這不是它要找的 `ids` 參數|
|`true`|`department=IT&ids=1&ids=2&ids=3`|✅ **成功！** .NET 看到重複的 `ids`，聰明地自動收集成 `List<int>`|

**使用時機判斷**：

- ✅ 該用：使用**平信格式**（未指定 contentType）+ 資料裡**包含陣列** + 後端是 .NET Framework 或 .NET Core 時，**必須**加 `traditional: true`
- ❌ 不用操心：沒有傳送陣列時，加不加都沒差
- ❌ 不用操心：**用 JSON 傳送時**（`contentType: 'application/json'` + `JSON.stringify`）。因為 traditional 只控制「物件轉平信」的規則，而 stringify 之後交給 jQuery 的已經是字串，根本不會啟動平信轉換機制，所以 traditional 設什麼都沒作用，後端也會靠 `[FromBody]` 順利接手

💣 **踩坑經驗**：很多新手發現後端 List 抓不到資料時，會慌亂地去改後端的 `[FromQuery]`、`[FromForm]`，甚至懷疑自己 C# 寫錯。其實只要打開 F12 Network 看一眼 Payload，如果變數名稱後面拖著醜醜的 `%5B%5D`（這就是 `[]` 的 URL 編碼），就代表你忘記設定 `traditional: true` 了！

## 2. .NET 後端接收：Model Binding 與四大指示詞

### 2.1 什麼是 Model Binding？

包裹送到 .NET 伺服器後，如果什麼工具都不用，工程師必須寫很底層的程式碼，把網路傳來的 0 和 1 慢慢轉成文字或檔案。為了解決這個痛苦，.NET 推出了超強機制「**模型繫結 (Model Binding)**」——它的工作就是「自動看懂包裹標籤，並把內容物塞進你寫好的 C# 變數裡」。

為了讓它精準工作，有時需要給它一些「指示詞（Attributes）」，也就是寫在 C# 參數前面的中括號 `[ ]`。

### 2.2 接收三種包裹的方式

**接收平信（form-urlencoded）**：

```csharp
[HttpPost]
public IActionResult SimpleForm(string username, string email, int age)
{
    // .NET Core 會自動進行 Model Binding
    // 只要前端傳的 name 跟後端參數名稱一樣，就會自動對應
    Console.WriteLine($"使用者: {username}");
    Console.WriteLine($"信箱: {email}");
    Console.WriteLine($"年齡: {age}");

    return Json(new { success = true });
}
```

**接收 JSON**（🚨 必須加 `[FromBody]`）：

```csharp
public class UserModel
{
    public string Username { get; set; }
    public string Email { get; set; }
    public int Age { get; set; }
}

[HttpPost]
public IActionResult ReceiveJson([FromBody] UserModel model)
{
    // [FromBody] 告訴 .NET Core 從 JSON 解析
    Console.WriteLine($"使用者: {model.Username}");
    Console.WriteLine($"信箱: {model.Email}");
    Console.WriteLine($"年齡: {model.Age}");

    return Json(new { success = true });
}
```

**接收檔案（便當盒）**：

```csharp
[HttpPost]
public IActionResult UploadFile(string description, IFormFile uploadFile)
{
    Console.WriteLine($"描述: {description}");

    if (uploadFile != null && uploadFile.Length > 0)
    {
        Console.WriteLine($"檔案名稱: {uploadFile.FileName}");
        Console.WriteLine($"檔案大小: {uploadFile.Length} bytes");
        Console.WriteLine($"檔案類型: {uploadFile.ContentType}");

        var filePath = Path.Combine("uploads", uploadFile.FileName);
        using (var stream = new FileStream(filePath, FileMode.Create))
        {
            uploadFile.CopyTo(stream);
        }

        return Json(new { success = true, fileName = uploadFile.FileName });
    }

    return Json(new { success = false, message = "沒有選擇檔案" });
}
```

便當盒的接收原理：.NET 確認包裹是便當盒後，會把裡面「每一個分隔層」都打開來看。只要前端 FormData 裡放入的名稱（`formData.append('username', 'john')`）剛好對應到後端 C# 參數的名稱（`string username`），就會自動把那個夾層的資料塞進變數。而只要參數型別寫 `IFormFile`，.NET 一看就知道這是檔案，會自動從便當盒裡把它拿出來。

### 2.3 💣 世紀大誤會：[FromBody] 不是「從 Body 拿資料」

很多人（包括很多資深工程師）直覺認為：「平信和 JSON 都裝在 HTTP Body 裡，那 `[FromBody]` 應該都能用吧？」——**錯！這是微軟的命名陷阱。**

**背後的脈絡**：對 .NET 來說，HTTP Body 確實是那個「大紙箱」，但 .NET 拆包裹時，不只是看箱子在哪裡，還要挑選適合的「**翻譯機（Formatter）**」來解讀內容：

- **[FromForm]（平信翻譯機）**：真實指令是「去紙箱裡，用『拆解 a=1&b=2 這種鍵值對』的方法把資料拿出來」
- **[FromBody]（JSON 翻譯機）**：真實指令是「去紙箱裡，拿『JSON 反序列化工具（強大的 3D 掃描器）』，把整包文字轉成複雜的 C# 物件」

💡 **海關比喻**：貨櫃（HTTP Body）裡裝了一疊「報關文件（平信表單）」。叫海關拿 `[FromForm]`（文件閱讀器），能輕鬆讀懂歸檔；但叫海關啟動 `[FromBody]`（X光斷層掃描儀 / JSON 解析器）去掃那疊紙，機器一掃發現這根本不是預期的複雜 3D 結構，就直接亮紅燈報錯（400 Bad Request）。

🚨 很多資深工程師都抱怨過：微軟當初如果把它命名為 `[FromJson]`，全世界大概會少踩一半的坑 😂。**[FromBody] 的重點不在於「資料在哪裡」，而是它強制規定了「必須使用 JSON 翻譯機來拆解這包資料」。**

**經典踩坑：前端傳 JSON、後端忘記加 [FromBody]**

```csharp
// ❌ 錯誤寫法：沒有指示詞
[HttpPost]
public IActionResult SaveUser(UserModel user)
```

.NET 會把它當成「預設的平信」處理，試圖去信封上找 `username=john`，結果找不到（因為你傳的是 JSON），最後 `user` 變數就是 `null`。程式繼續往下執行，就會發生可怕的 `NullReferenceException`。

```csharp
// ✅ 正確寫法：加入 [FromBody] 指示詞
[HttpPost]
public IActionResult SaveUser([FromBody] UserModel user)
```

**反向踩坑也一樣致命**：「只要是 POST 就一定要加 [FromBody]」是致命的誤會！如果前端是把資料轉成「平信」裝進箱子，你在後端加了 `[FromBody]`，.NET 就會拿拆 JSON 的工具去拆平信，直接報錯當機。

### 2.4 網址上的資料：[FromRoute] 與 [FromQuery]

GET（明信片）的資料放在網址上，但網址其實分成兩個存放資料的區域。拿一個常見的 API 網址來解剖：

```
https://asus.com/api/Employee/IT/Detail?id=123
```

- **區域 A：路徑 (Route)** → `/Employee/IT/Detail`
    - 比喻：檔案的「資料夾路徑」，或「信封上的實體地址（台北市/中和區/建一路）」
    - 這裡的 `IT` 是可以變動的資料（可換成 HR、RD），用 `[FromRoute]` 去抓
- **區域 B：查詢字串 (Query)** → `?id=123`
    - 比喻：信封角落的「補充備註（分機號碼：123）」，特徵是永遠接在 `?` 後面，用 `&` 串聯
    - 用 `[FromQuery]` 去抓

```csharp
// 告訴系統，網址結構是 api/Employee/{變數}/Detail
[HttpGet("api/Employee/{department}/Detail")]
public IActionResult GetDetail(
    [FromRoute] string department,  // 從路徑抓出 "IT"
    [FromQuery] int id              // 從 ? 後面抓出 123
)
{
    Console.WriteLine($"部門: {department}, 員工代號: {id}");
    return Json(new { success = true });
}
```

**實戰練習**：GTS 系統有個下載報表的 API：`https://gts.asus.com/api/report/2025/download?format=pdf`（2025 是年份、pdf 是格式）。接收時就是：

```csharp
[FromRoute] int year      // 2025 藏在網址的實體路徑裡
[FromQuery] string format // format=pdf 是問號後面的補充條件
```

### 2.5 四大 [FromX] 指示詞終極對比表

|指示詞|📦 資料在哪裡？|💡 什麼時候該用？|🚫 常見踩坑|
|---|---|---|---|
|`[FromRoute]`|網址的路徑裡<br>`/user/123`|參數是用來「定位某個具體資源」時（特定員工代號、訂單編號）|網址設定忘記加上 `{變數}` 佔位符|
|`[FromQuery]`|網址的 `?` 後面<br>`?sort=desc&page=2`|「額外的篩選條件、排序、分頁」|把極機密資料（如密碼）放這裡，會被看光光|
|`[FromForm]`|Body（平信/便當盒）|傳統表單送出、檔案上傳|忘記在前端設定 FormData|
|`[FromBody]`|Body（JSON）|前端傳送複雜的 JSON 結構時|前端忘記用 `JSON.stringify()`，導致解析失敗|

> 📌 **口訣**：資料在 Body，找 Form 或 Body；資料在網址，用 Route 抓路徑，用 Query 抓問號後面的條件。
> 
> 另外補充：雖然平信和便當盒「什麼都不寫」.NET 也能自動猜到要去表單裡找，但實務上為了讓程式碼更明確，我們有時會主動加上 `[FromForm]`，像在參數前掛個小牌子，明確告訴系統和同事：「這個變數，請去『表單』的夾層裡面找喔！」

### 2.6 [FromQuery] 可以不寫嗎？——談 Model Binding 的預設搜尋順序

看了上面的範例，你可能會有個疑問：「既然是 HttpGet，Model Binding 時系統本來就會去網址找 `?` 後面的參數，那 `[FromQuery]` 還有沒有需要寫？」

答案是：**技術上可以不寫，程式照樣會動**——但實務上建議寫。我們來拆解背後的機制。

**背後的脈絡：預設的「口袋搜尋順序」**

當參數前面什麼指示詞都沒掛時，.NET 的模型繫結不會只找一個地方，它會拿著參數名稱，按照固定順序把幾個口袋都摸一遍：

```
表單資料 (Form) → 路由值 (Route) → 查詢字串 (Query String)
```

以 `GetDetail(string department, int id)` 完全不掛指示詞、網址 `api/Employee/IT/Detail?id=123` 為例：

1. `department` → Form 沒有（GET 沒有 Body）→ **Route 找到 "IT"** ✅
2. `id` → Form 沒有 → Route 沒有 → **Query 找到 123** ✅

所以不寫也能綁定成功。這跟前面講的觀念一致：`[FromForm]` 也是「可寫可不寫」，平信和便當盒靠預設機制就能收——真正**非寫不可**的只有 `[FromBody]`（因為 JSON 翻譯機預設不會啟動）。

**那為什麼實務上還是建議寫？**

- **理由一：掛小牌子，程式碼自我說明。** 跟 `[FromForm]` 一樣的道理——同事（包括三個月後的你）一眼就知道「這個參數是從問號後面來的」，不用去猜預設搜尋順序。
- **理由二（真正關鍵）：鎖定來源，避免撞名時被「劫走」。** 預設搜尋順序中 **Route 的優先權高於 Query**。想像哪天路由設計改版，變成 `api/Employee/{department}/Detail/{id?}`——這時 `id` 同時存在於路由和查詢字串的可能性出現了，不掛指示詞的話，Route 會先搶走綁定權，你的 `?id=123` 就默默失效，而且**不會報錯**，只是拿到不對的值，這種 bug 抓起來最痛苦。掛上 `[FromQuery]` 就是明確下令：「只准去問號後面拿，別的地方的同名資料一律不理。」
- **【補充說明】理由三：如果 Controller 有掛 `[ApiController]`**（純 Web API 專案常見），框架會自動推斷：出現在路由樣板裡的參數視為 `[FromRoute]`、其他簡單型別視為 `[FromQuery]`、複雜型別視為 `[FromBody]`。這種專案裡不寫也很安全——但明寫依然是主流的團隊風格。

> 📌 **小結**：`[FromRoute]` / `[FromQuery]` / `[FromForm]` 都屬於「不寫能動、寫了更穩」的保險絲；只有 `[FromBody]` 是「不寫就掛」的必需品。

### 2.7 冷知識：GET 改 POST（不轉 JSON），後端要改多少？

**答案：只需要把 `[HttpGet]` 改成 `[HttpPost]`，接收參數的程式碼完全不用改！**

為什麼這麼神奇？因為前端只是單純把 `type: 'GET'` 改成 `type: 'POST'`（data 保持物件、不用 stringify）時，jQuery 秘書照樣把物件壓扁成平信 `id=1`，塞進 POST 紙箱，貼上平信標籤送出。後端看到平信標籤，就啟動「平信處理機制」：

|運送方式|前端傳送的真實模樣|後端 .NET Core 接收寫法|.NET 的底層拆解動作|
|---|---|---|---|
|GET（明信片）|網址列：`/api/GetWorker?id=1`|`[HttpGet]`<br>`public IActionResult GetWorker(int id)`|從**網址列**上找到 id，塞進變數|
|POST（紙箱裝平信）|Body 裡面裝著：`id=1`|`[HttpPost]`<br>`public IActionResult GetWorker(int id)`|從**紙箱 (Body) 裡的平信**找到 id，塞進變數|

只要前端沒轉 JSON（送出的是平信），不管你是用 GET 放網址，還是用 POST 放 Body，.NET 都能靠著相同的變數名稱自動對應好。

## 3. 後端回傳資料（Response）

### 3.1 回傳方向的角色互換

現在包裹寄送的方向反過來了：**.NET 伺服器變成「寄件人」，瀏覽器（和裡面的 jQuery）變成「收件人」**。回傳包裹上的 Content-Type 標籤，由**後端**決定並貼上。

**實務上解決什麼問題？** 如果 Portal 系統後端產生一份「員工清單 Excel 檔」回傳，但包裹忘記貼標籤或貼錯成「純文字」，瀏覽器會心想：「喔，這是純文字文章，我直接顯示在螢幕上吧！」結果使用者畫面上印出滿滿幾萬字的亂碼（二進位檔案的文字呈現），而不是預期中的下載視窗 😱。

後端貼的 Content-Type，就是給瀏覽器的「**包裹拆解行動指令**」：

- 看到 `text/html` → 瀏覽器：「我要把它畫成漂亮的網頁畫面」
- 看到 `application/pdf` → 瀏覽器：「我要啟動 PDF 閱讀器，或跳出下載視窗」
- 看到 `application/json` → jQuery：「我要啟動 JSON 翻譯蒟蒻，把它還原成程式可讀的立體物件」

### 3.2 .NET 的回傳方法（Helper Methods）：讓框架幫你貼標籤

好消息：在 .NET 裡幾乎不需要手動寫標籤字串。呼叫對應的方法，.NET 底層就會自動包裝並精準貼上正確標籤：

|情境|.NET 後端寫法|自動貼上的 Content-Type|瀏覽器的反應動作|
|---|---|---|---|
|回傳網頁畫面|`return View();`|`text/html`|解析 HTML 並渲染成視覺網頁|
|回傳 API 資料|`return Json(new { name = "john" });`|`application/json`|交給前端 JavaScript 處理資料|
|回傳特定檔案|`return File(位元組, "application/pdf");`|`application/pdf`（需手寫標籤告訴它檔案類型）|開啟 PDF 或跳出下載視窗|
|強制下載檔案|`return File(位元組, "application/octet-stream");`|`application/octet-stream`|強制彈出「另存新檔」視窗|

各種回傳的完整範例：

```csharp
// 回傳 View（自動設定 text/html）
public IActionResult Dashboard()
{
    return View();
}

// 回傳 HTML 片段
public IActionResult GetUserCard(int userId)
{
    var html = $@"
        <div class='user-card'>
            <h3>使用者 {userId}</h3>
            <p>部門：IT</p>
        </div>";

    return Content(html, "text/html");
}

// 簡單 JSON 回應
public IActionResult GetUserList()
{
    var users = new[] {
        new { id = 1, name = "John", department = "IT" },
        new { id = 2, name = "Jane", department = "HR" }
    };

    return Json(users); // Content-Type: application/json
}

// Excel 檔案（使用 ClosedXML）
public IActionResult ExportToExcel()
{
    using var workbook = new XLWorkbook();
    var worksheet = workbook.Worksheets.Add("資料");

    worksheet.Cell(1, 1).Value = "姓名";
    worksheet.Cell(1, 2).Value = "部門";
    worksheet.Cell(2, 1).Value = "John";
    worksheet.Cell(2, 2).Value = "IT";

    // 自動調整欄寬
    worksheet.ColumnsUsed().AdjustToContents();

    using var stream = new MemoryStream();
    workbook.SaveAs(stream);
    var excelBytes = stream.ToArray();

    return File(excelBytes,
        "application/vnd.openxmlformats-officedocument.spreadsheetml.sheet",
        "資料.xlsx");
}

// 強制下載任何檔案
public IActionResult ForceDownload(string fileName)
{
    var filePath = Path.Combine("files", fileName);
    if (!System.IO.File.Exists(filePath))
    {
        return NotFound();
    }

    var fileBytes = System.IO.File.ReadAllBytes(filePath);
    return File(fileBytes, "application/octet-stream", fileName);
}
```

### 3.3 🚨 經典踩坑：假裝是純文字的 JSON

剛學 .NET 後端的人超常犯：想回傳 JSON 時，自己手動把字串拼出來，然後用回傳純文字的 `Content()` 方法：

```csharp
// ❌ 錯誤示範：用純文字方法回傳 JSON 字串
public IActionResult GetUser()
{
    var jsonString = "{\"name\":\"john\", \"age\":30}";
    return Content(jsonString, "text/plain"); // 錯誤！標籤貼成純文字了
}
```

**慘劇**：前端 jQuery 收到包裹，一看標籤寫 `text/plain`，就想：「喔，這是一段普通的文字。」於是不會幫你轉換成 JavaScript 物件。當你在前端寫 `console.log(response.name)` 時，就會跑出 `undefined`，因為它還只是一段死板的字串！

```csharp
// ✅ 正確示範：直接把 C# 物件丟給 Json() 方法
public IActionResult GetUser()
{
    var userObj = new { name = "john", age = 30 };
    return Json(userObj); // .NET 自動轉成 JSON 字串，並貼上 application/json 標籤
}
```

> 📌 **小結**：伺服器回傳時，不用自己手作包裝。善用 .NET 內建的 `Json()`、`File()`、`View()` 等方法，標籤就不會貼錯。

### 3.4 前端接收回應：jQuery 的自動解析機制

jQuery 會根據以下**順序**決定如何處理回應：

1. **你明確指定的 `dataType`**（最高優先級）
2. **伺服器回應的 `Content-Type`**（如果沒設定 dataType，就自動從 Response Header 的 Content-Type 判斷）
3. **內容格式的智慧判斷**

**範例一：JSON 回應（自動解析成物件）**

伺服器端：

```csharp
[HttpPost]
public IActionResult SaveUser(UserModel user)
{
    return Json(new {
        success = true,
        message = "儲存成功",
        userId = 123
    });
    // 自動設定 Content-Type: application/json
}
```

前端接收：

```javascript
$.ajax({
    url: '/api/saveUser',
    type: 'POST',
    data: { username: 'john', age: 30 },
    success: function(response) {
        // jQuery 一看標籤是 application/json，就啟動內建的
        // 「JSON 翻譯蒟蒻（JSON.parse）」，自動解析成物件
        console.log(typeof response);   // "object"
        console.log(response.success);  // true
        console.log(response.message);  // "儲存成功"

        if (response.success) {
            alert('儲存成功！使用者 ID：' + response.userId);
        }
    }
});
```

這就是為什麼你在 `success: function(response)` 裡，可以直接用 `.` 呼叫屬性而不會報錯。

**範例二：HTML 片段回應（保持字串）**

伺服器端：

```csharp
public IActionResult GetUserCard(int userId)
{
    var html = $@"
        <div class='user-card'>
            <h3>使用者 {userId}</h3>
            <p>部門：IT</p>
            <button onclick='editUser({userId})'>編輯</button>
        </div>";

    return Content(html, "text/html");
}
```

前端接收：

```javascript
$.ajax({
    url: '/api/getUserCard',
    data: { userId: 123 },
    success: function(response) {
        // 🚨 $.ajax() 對 text/html 的處理是：不會自動解析成 DOM 物件，而是保持為字串
        console.log(typeof response);  // "string"

        // 直接插入到頁面中，jQuery 會自動解析 HTML
        $('#userContainer').html(response);
        // 等同於：$('#userContainer').get(0).innerHTML = response;
    }
});
```

### 3.5 Ok() 與 BadRequest()：標籤一樣，命運不同【補充說明】

實戰程式碼裡（例如前面 1.8 的 `UpdateProfile`）你會看到 `return Ok(...)` 和 `return BadRequest(...)`，它們回傳的 Content-Type 是什麼？

答案：當你把「物件」塞給 `Ok()` 或 `BadRequest()` 時，.NET Core 會啟動「輸出格式化器（Output Formatter）」把物件序列化，**預設就是 JSON**，所以貼的標籤跟 `Json()` 一模一樣：

```
Content-Type: application/json; charset=utf-8
```

它們的差別其實不在「標籤」，而在 **HTTP 狀態碼**——你可以想像成包裹上除了內容物標籤，還有一張「配送結果單」：

|寫法|狀態碼|Content-Type|前端 jQuery 走哪條路|
|---|---|---|---|
|`return Json(obj)`|200 OK|`application/json`|`success` / `.done()`|
|`return Ok(obj)`|200 OK|`application/json`|`success` / `.done()`|
|`return BadRequest(obj)`|**400** Bad Request|`application/json`|🚨 `error` / `.fail()`|

**`new { success = true, message = "資料更新成功" }` 到前端變什麼樣子？**

HTTP Body 裡是這串 JSON 字串：

```json
{"success":true,"message":"資料更新成功"}
```

jQuery 一看標籤是 `application/json`，就啟動翻譯蒟蒻自動 parse，所以在 `.done()` 裡可以直接寫：

```javascript
.done(function(response) {
    console.log(response.success);  // true（注意：是布林值，不是字串 "true"）
    console.log(response.message);  // "資料更新成功"
});
```

**🚨 坑一：BadRequest 的資料不會進 success！**

因為狀態碼是 400，jQuery 會判定「請求失敗」，直接跳進 `.fail()`。這時後端塞的那包 JSON 要從 `jqXHR` 身上撈：

```javascript
.fail(function(jqXHR, textStatus, errorThrown) {
    // jqXHR.responseJSON → jQuery 幫你 parse 好的物件（回應是 JSON 時才有）
    console.log(jqXHR.responseJSON.message);
    // jqXHR.responseText → 原始字串版本
});
```

很多新手在 `success` 裡等 BadRequest 的錯誤訊息等到天荒地老，就是不知道它走的是另一條路。

**🚨 坑二：大小寫會變！**

上面的匿名物件因為屬性本來就是小寫（`success`、`message`），所以看不出來。但如果回傳的是正式的 C# 類別（PascalCase 命名）：

```csharp
public class ResultModel
{
    public bool Success { get; set; }     // C# 慣例：大寫開頭
    public string Message { get; set; }
}
return Ok(new ResultModel { Success = true, Message = "OK" });
```

.NET Core 3.1 預設的序列化器（System.Text.Json）會自動套用 **camelCase 命名策略**，前端收到的是：

```json
{"success":true,"message":"OK"}
```

所以前端要寫 `response.success`（小寫），寫 `response.Success` 會拿到 `undefined`。這個「後端大寫、前端小寫」的落差，是 Portal 這種 .NET Core 專案的經典地雷；而 GTS（.NET Framework 4.8 MVC 的 `Json()`）預設**不會**轉小寫，會照 C# 屬性原樣輸出 `Success`——兩套系統行為不同，切換維護時要特別留意。

### 3.6 submit vs $.ajax()：後端回傳要跟前端「配對」【補充說明】

接續上一節，有個非常關鍵的延伸問題：**如果前端是用傳統 `<form>` submit 而不是 `$.ajax()`，`return Ok()` / `return BadRequest()` 也是回 `application/json` 嗎？回到前端會怎麼跑？**

**第一半：標籤不變。** 後端回傳什麼 Content-Type，是由**後端的程式碼**決定的。後端根本不知道、也不在乎這個請求是秘書（$.ajax）發的，還是郵差（表單 submit）親自送來的——它只管照自己的程式碼打包、貼標籤、寄出。所以不管前端怎麼發，回傳的包裹都是 `application/json`。

**第二半：收件人變了，命運就完全不同。** 傳統 `<form>` submit 是一次「**整頁導航**」——等於瀏覽器帶著整個視窗離開現在的頁面，前往 action 網址，伺服器回什麼，瀏覽器就把「整個畫面換成那個東西」。接著瀏覽器照老規矩「盲從標籤」：一看包裹貼著 `application/json`，心想「這不是 text/html 網頁設計圖，我不用啟動畫筆排版……那就當純文字照抄吧」。

於是使用者會看到——**原本漂亮的表單頁面整個消失，整個瀏覽器畫面變成白底黑字的一行字**：

```
{"success":true,"message":"資料更新成功"}
```

網址列也變成了 action 的網址。這就是實務上超級經典的 bug 現場：「**為什麼我按送出之後，畫面變成一串 JSON？**」（新版 Chrome/Edge 可能會好心地幫 JSON 排版上色，但本質一樣——頁面已經被整頁換掉了。）

**BadRequest (400) 呢？** 幾乎一樣慘：狀態碼只是「配送結果單」，瀏覽器照樣把 Body 顯示出來，使用者一樣看到整頁 JSON 文字，只是換成錯誤訊息版本——不會跳 alert、不會留在原頁標紅錯誤欄位。還有隱藏傷害：因為是整頁導航，**使用者剛剛辛苦填的表單內容全部消失了**。

**結論：前端的發送方式，要和後端的回傳方式配對**：

|前端發送方式|後端該配的回傳|理由|
|---|---|---|
|傳統 `<form>` submit|`return View(...)` 或 `return RedirectToAction(...)`|郵差收件，要嘛給它一張新網頁畫，要嘛叫它轉往別頁（成功後轉跳結果頁，最傳統的 MVC 流程）|
|`$.ajax()`|`return Json(...)` / `Ok(...)` / `BadRequest(...)`|秘書收件，JSON 進記憶體，`.done()` 裡優雅地更新畫面局部，**頁面不重刷**|
|傳統 submit / location.href|`return File(...)`|唯一的例外組合！郵差看到檔案標籤會跳下載視窗——就是第 4 章的「隱藏表單下載檔案」招式|

傳統 submit + Redirect 的組合在業界有個名字叫 **PRG 模式（Post-Redirect-Get）**：POST 成功後不直接回傳畫面，而是 Redirect 讓瀏覽器再用 GET 去拿結果頁，這樣使用者按 F5 重新整理時才不會跳出「要重新提交表單嗎？」並意外重複送單。在 GTS 這種較傳統的 .NET Framework MVC 系統裡，Controller 結尾常見的 `RedirectToAction("Index")` 就是這個道理。

> 📌 **一句話總結**：`Ok()` / `BadRequest()` 回的永遠是 `application/json`，不管誰來拿；但拿的人如果是郵差（submit），它只會把 JSON 當純文字整頁貼出來。**JSON 回應是為秘書（AJAX）設計的，submit 請配 View 或 Redirect。**

## 4. 檔案下載：AJAX 的死胡同與三大招式

### 4.1 為什麼 AJAX 下載檔案「看起來像當機」？

**回傳 `application/octet-stream`（檔案）時，是由誰處理的？**

如果你用傳統的 HTML `<form>` 或直接點擊 `<a>` 連結，瀏覽器（郵差）會**親自接手包裹**——看到檔案標籤，立刻彈出「另存新檔」視窗。

但如果你用 `$.ajax()`？AJAX 就像你派出去的「**秘密特務**」🕵️‍♂️：特務跑到伺服器把檔案拿回來了，但他會把這包二進位檔案資料**死死捏在自己手裡**（存放在瀏覽器的記憶體變數中），然後交給你的 `success` 函數。因為這一切都在背景「偷偷」發生，瀏覽器根本不知道有檔案下載這回事，所以**絕對不會跳出下載視窗**！

秘書（jQuery）看著手上這包二進位檔案，根本沒有權限也沒有能力去呼叫作業系統的「另存新檔」視窗，只能默默把它抓在手上（佔用記憶體），然後……就沒有然後了。使用者的畫面看起來就像「按了按鈕卻毫無反應」，這就是 AJAX 下載檔案最常見的死胡同。

|請求方式|負責接收的對象|遇到檔案標籤時的反應|
|---|---|---|
|網址列 / `<form>` 送出 / `<a>` 連結|瀏覽器本人 🌐|✅ 成功！自動跳出「另存新檔」視窗 💾|
|`$.ajax()` 背景送出|jQuery 秘密特務 🕵️‍♂️|❌ 失敗！檔案留在記憶體裡，畫面毫無反應，看起來像當機|

> 📌 **鐵律：永遠不要用單純的 AJAX 來下載檔案。**

### 4.2 招式一：location.href（公司系統目前的主流做法）

`@Url.Action` 是 .NET MVC 的語法，最後會變成一段網址，例如 `/Export/EmployeeList?dept=IT`：

```javascript
// 告訴瀏覽器，請直接前往這個網址！
window.location.href = '/Export/EmployeeList?dept=IT';
```

**背後脈絡**：這等於放棄用 jQuery 秘書，直接叫老闆（瀏覽器視窗）去那個網址。因為是瀏覽器親自出馬，當伺服器回傳 Excel 標籤時，瀏覽器立刻知道：「這是檔案，我要跳出下載視窗！」

- 👍 **優點**：超級簡單，一行程式碼搞定
- 👎 **缺點與限制**：只能發送 **GET** 請求。如果要匯出「使用者勾選的 500 個員工 ID」，把 500 個 ID 塞進網址會太長而直接報錯（網址通常有 2048 字元的長度限制）

**附帶的使用者體驗問題**：如果伺服器要算 10 秒才能產出 Excel，用 location.href 這 10 秒內畫面完全無法顯示 Loading 動畫，看起來像卡住。

### 4.3 招式二：隱藏表單 Submit（Hidden Form）

如果參數很多、必須用 POST（把資料藏在 Request Body）怎麼辦？

**背後脈絡**：用 JavaScript 在畫面上偷偷做一個「看不見的傳統表單」，把幾百個 ID 塞進去，然後觸發原生表單的 `submit()`。因為這是**原生表單送出**，一樣是瀏覽器親自處理，所以能順利跳出下載。

- 👍 **優點**：可以用 POST 送大量條件，不受網址長度限制
- 👎 **缺點**：寫起來囉嗦，要在畫面上動手腳；現代前端開發較少用了

### 4.4 招式三：AJAX + Blob（現代的高級特務）

如果「真的很想用 AJAX」送出複雜的 JSON 條件，又想下載檔案：一樣讓秘書去拿檔案，但拿到之後，秘書在辦公室裡「偽造」一個下載按鈕，然後自己按下去。

```javascript
// 注意：因為要處理二進位，通常會改用原生的 fetch 或特別設定 AJAX
fetch('/api/export', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ ids: [1, 2, 3 /* ...500 */] })
})
.then(response => response.blob()) // 把秘書手上的資料轉成 Blob（大塊二進位物件）
.then(blob => {
    // 1. 秘書把手上的檔案變成一個「臨時網址」
    var url = window.URL.createObjectURL(blob);
    // 2. 秘書做一個隱藏的超連結
    var a = document.createElement('a');
    a.href = url;
    a.download = '員工名單.xlsx'; // 指定檔名
    // 3. 秘書偷偷幫你點擊這個連結，觸發瀏覽器的下載！
    document.body.appendChild(a);
    a.click();
    a.remove(); // 用完就銷毀
});
```

- 👍 **優點**：可以用 AJAX（能顯示 Loading 動畫！）、可以用 POST、可以傳 JSON，目前最完美的體驗
- 👎 **缺點**：程式碼比較複雜，還要自己處理記憶體釋放的問題

### 4.5 檔案下載招式對比總結

|方法|發送方式|什麼時候該用？（判斷力）|什麼時候不該用？|
|---|---|---|---|
|`location.href`|GET|**最常用！** 匯出條件不多時（例如只傳日期區間或部門代碼），用這招最快|條件複雜、參數極多、或想顯示 Loading 圈圈時|
|隱藏表單 Submit|POST|需要傳遞大量表單資料，但團隊還不熟 Blob 語法時的傳統解法|現代前端開發較少用了，稍嫌麻煩|
|AJAX + Blob|POST (JSON)|**進階首選。** 需要傳遞複雜 JSON，且希望有完整錯誤處理和 Loading 動畫時|只是單純下載一個固定檔案時（殺雞焉用牛刀 🔪）|

實務原則：如果系統只是單純下載一份員工名單，`location.href` 一行搞定就好。Blob 雖然強大，但程式碼長、維護難度高——等哪天真的遇到「必須勾選 500 個複雜條件才能下載報表」的需求，腦中有「該請出 Blob 了」的觀念，再去查語法都來得及。

## 5. F12 開發者工具：包裹追蹤系統與 X 光機 🔍

### 5.1 實務上解決什麼問題？

寫程式一定會遇到 Bug。前後端最常發生的爭執：

- 前端：「我明明就有傳資料給你啊！」
- 後端：「可是我這邊收到的就是空的啊！」

為了解決這種「各說各話」的通靈狀況，我們必須有一個「絕對客觀的第三方證據」——按下 **F12** 打開瀏覽器的「開發者工具」，它就像網頁的「行車紀錄器」。

### 5.2 F12 四大常用標籤

|F12 標籤名稱|📦 生活比喻|實務上解決什麼問題？|
|---|---|---|
|Elements（元素）|檢視網頁的外觀與結構|檢查按鈕為什麼跑版、字體顏色對不對（HTML/CSS）|
|Console（主控台）|秘書（JavaScript）的日誌本|檢查前端程式碼有沒有寫錯、變數有沒有抓到、看 `console.log` 印出的東西|
|🚨 **Network（網路）**|**包裹追蹤與 X 光機**|看清楚郵差到底寄了什麼包裹出去、拿了什麼包裹回來|
|Sources（原始碼）|存放原始檔案的倉庫|看瀏覽器載入了哪些 JS、CSS 檔案，或在這裡下中斷點除錯|

### 5.3 Network 標籤怎麼看？

點開 Network 後，在網頁上點擊按鈕、發出 AJAX 請求，清單上就會多出一行紀錄。點擊那行紀錄，右邊詳細面板有三個最重要的分頁：

1. **Headers（標頭）**：「包裹的外箱標籤」。往下滑找到 **Request Headers**，就能親眼看到瀏覽器有沒有幫你貼上 `Content-Type: application/json` 或 `multipart/form-data`。
2. **Payload（傳載資料 / 請求主體）**：「包裹裡面的真實內容」。你的密碼、JSON 字串、便當盒夾層（FormData）到底長什麼樣子，是不是跟後端預期的一樣？
3. **Response（回應）**：「後端退回來的信」。如果後端報錯（400 Bad Request 或 500 Internal Server Error），點開這裡通常能看到真正的死因（例如「找不到 username 參數」或「反序列化失敗」）。

### 5.4 前輩的防坑小技巧

**技巧 1：過濾出秘書的包裹（Filter by Fetch/XHR）**

很多新手一打開 Network，被裡面幾十上百條的圖片、CSS 紀錄嚇到，找不到自己的 AJAX。Network 面板上方有一排篩選按鈕（All, Fetch/XHR, JS, CSS...），**點擊 Fetch/XHR**，畫面就只會顯示由 AJAX（秘書）發出的請求，瞬間乾淨清爽！

**技巧 2：勾選 Preserve log（保留紀錄）**

有時按下「送出表單」，網頁整頁重新載入（閃一下），Network 預設會把舊紀錄清空，你就什麼死因都看不到了！勾選 **Preserve log**，就能把跨頁面的請求都保留下來——除錯時的救命稻草。

### 5.5 實戰診斷：一眼抓出「忘記 JSON.stringify」

同事說要傳 JSON，但忘記 `JSON.stringify()`，後端收不到資料報錯。打開 F12 → Network → 點失敗的請求 → 切到 **Payload**，用這張表診斷：

|同事的寫法|📦 Payload（X光機看到的真實內容）|🕵️‍♂️ 你的診斷結果|
|---|---|---|
|❌ 忘記轉字串（簡單物件）|`name=john&age=18`|**被轉成平信了！** 標籤貼 JSON 但內容是 urlencoded 格式，`{}` 大括號不見了，後端一定解析失敗|
|❌ 忘記轉字串（複雜物件）|`name=john&preferences=[object+Object]`|**抓到了！** 出現 `[object Object]`，代表 jQuery 強制壓扁子物件失敗，資料已經壞掉|
|✅ 正確寫法 `JSON.stringify(...)`|`{"name":"john", "preferences":{"theme":"dark"}}`|**完美！** 有完整的 `{}` 和 `""` 雙引號，這才是標準的 JSON 說明書|

> 📌 **小結**：發送 JSON 的請求中，Payload 裡沒看到完整的 `{}` 結構，而是看到 `&` 符號串接或詭異的 `[object Object]`——不用懷疑，百分之百是忘記寫 `JSON.stringify()`！另外，如果變數名稱後面拖著 `%5B%5D`（`[]` 的 URL 編碼），就是忘了 `traditional: true`。

## 本篇總結：三大神兵利器 + 完整拼圖

1. ✉️ **平信信封（x-www-form-urlencoded）**
    - 何時用：傳統表單送出、簡單的 AJAX 文字傳輸
    - 怎麼做：前端預設就是它；後端什麼都不用寫（或加 `[FromForm]`）就能收；**平信 + 陣列 + .NET 記得加 `traditional: true`**
2. 🍱 **分隔便當盒（multipart/form-data）**
    - 何時用：只要有「檔案（大頭照、Excel）」就必用！
    - 怎麼做：前端用 `new FormData()` 打包，AJAX 設 `contentType: false` 和 `processData: false` 讓郵差（瀏覽器）接手；後端用 `IFormFile` 收檔案
3. 📖 **JSON 說明書（application/json）**
    - 何時用：傳遞多層級、複雜的結構化資料（陣列、物件）
    - 怎麼做：前端必須貼標籤 `contentType: 'application/json'`，且內容「絕對要」用 `JSON.stringify()` 壓扁；後端「絕對要」加 `[FromBody]` 才能正確呼叫翻譯蒟蒻

加上後端智慧拆箱（`[FromBody]` / `[FromForm]` / `[FromRoute]` / `[FromQuery]`）、檔案下載交給瀏覽器（`location.href` / 隱藏表單 / Blob），以及 **F12 Network 標籤** 這面照妖鏡——以後遇到任何前後端傳輸問題，你都有能力一眼看穿！

### 最重要的實務原則（貼在螢幕旁邊）

1. **有檔案上傳就用 multipart/form-data**，不能用 JSON
2. **AJAX 發送 JSON 要用 JSON.stringify()**
3. **檔案上傳的 AJAX 要設定 contentType: false（+ processData: false）**
4. **遇到問題先檢查 F12 Network 標籤**
5. **後端正確設定回應的 Content-Type**
6. **前端發送方式要和後端回傳配對**：submit 配 View / Redirect；$.ajax 配 Json / Ok / BadRequest；檔案下載交給瀏覽器 + File
