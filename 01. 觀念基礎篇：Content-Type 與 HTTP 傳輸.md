# 01. 觀念基礎篇：Content-Type 與 HTTP 傳輸

> **閱讀順序**：本篇是三部曲的第一篇。請先讀完本篇，再依序閱讀《02. 前後端實戰篇》與《03. $.ajax() 完整範本》。

---

## 1. 為什麼需要 Content-Type？

### 1.1 實務上解決的問題

想像一下，你寄了一個沒有任何標記的棕色紙箱 📦 給同事，裡面裝著一個精美的草莓蛋糕。同事收到後根本不知道裡面是什麼，可能直接把它塞進公文櫃，或者倒過來放，蛋糕就毀了。

伺服器也是一樣：如果前端丟了一大包資料過去，卻沒說是什麼格式，伺服器就會用錯方式去拆解（例如把圖片當成純文字來讀），導致程式當機、出錯，或者根本收不到資料。

為了解決這個問題，寄件人必須在包裹外貼上明確的「分類標籤」🏷️，寫上「食品：請冷藏」或「文件：請保持平整」。在程式的世界裡，這個分類標籤就叫做 **Content-Type**。

### 1.2 技術定義

- Content-Type 是 **HTTP Header** 的一部分
- 用來描述 HTTP 訊息主體（Body）的資料格式
- 讓接收方知道如何正確處理資料

> 📌 **小結**：Content-Type 就是網路資料傳輸時，貼在資料包裹外的「格式說明標籤」，確保接收方能正確解讀內容。

### 1.3 誰負責貼標籤？

很多人以為要自己手動設定，其實大部分情況是**瀏覽器自動幫你貼**的。這也是為什麼許多開發者一開始根本沒意識到這個標籤的存在。

**請求時（Request）**：

```html
<!-- 瀏覽器發送表單時自動設定 -->
<form method="post" action="/submit">
    <input type="text" name="username" value="john" />
    <input type="text" name="age" value="45" />

    <!-- 瀏覽器自動設定 Content-Type: application/x-www-form-urlencoded -->
    <!-- HTTP Body => username=john&age=45 -->
</form>
```

```javascript
// 開發者也可以在 AJAX 中手動設定
$.ajax({
    contentType: 'application/json', // 我們告訴伺服器：我送的是 JSON
    data: JSON.stringify({ name: 'john' })
});
```

**回應時（Response）**：由後端決定並貼上。

```csharp
// 伺服器設定回應的 Content-Type: application/json
public IActionResult GetData()
{
    return Json(new { name = "john" }); // 伺服器告訴瀏覽器：我回傳的是 JSON
}
```

### 1.4 貼錯標籤會發生什麼事？（先給你看一眼災難現場）

**錯誤示範**：

```csharp
// 伺服器回傳 JSON 資料，但設定錯誤的 Content-Type
public IActionResult GetUser()
{
    var json = "{\"name\":\"john\",\"age\":30}";
    return Content(json, "text/plain"); // 錯誤：告訴瀏覽器這是純文字
}
```

```javascript
// 前端收到回應
$.ajax({
    url: '/GetUser',
    success: function(response) {
        console.log(typeof response); // "string" - 被當作文字處理
        console.log(response.name);   // undefined - 無法直接存取屬性

        // 必須手動解析
        var userData = JSON.parse(response);
        console.log(userData.name);   // "john"
    }
});
```

**正確示範**：

```csharp
public IActionResult GetUser()
{
    return Json(new { name = "john", age = 30 }); // Content-Type 自動設為 application/json
}
```

```javascript
$.ajax({
    url: '/GetUser',
    success: function(response) {
        console.log(typeof response); // "object" - jQuery 自動解析成物件
        console.log(response.name);   // "john" - 可以直接存取屬性
    }
});
```

這個例子的細節我們會在《02. 實戰篇》深入解釋，這裡你只要先感受到：**標籤貼錯，接收方就會用錯誤的方式拆包裹**。

## 2. 資料傳遞的兩大門派：GET vs POST

在講各種 Content-Type 之前，要先建立一個更底層的觀念：資料是「搭什麼交通工具」過去的。

### 2.1 生活比喻：明信片 ✉️ vs 密封紙箱 📦

**GET 就像「明信片」✉️**：

- 你把要傳遞的資訊，直接寫在明信片的地址（URL 網址）旁邊
- 特性：大家都看得到（直接顯示在瀏覽器網址列，如 `?name=john`），而且因為郵局對地址長度有限制，所以不能寫太多字（網址通常有 2048 字元的長度限制）

**POST 就像「密封紙箱」📦**：

- 你把資訊塞在紙箱內部，也就是 **HTTP 主體（HTTP Body）** 裡
- 特性：外觀看不出裡面裝什麼，容量非常大，適合裝複雜的資料或檔案

### 2.2 GET 方法的特性

```html
<!-- GET 方法的表單 -->
<form method="get" action="/search">
    <input type="text" name="keyword" value="javascript" />
    <input type="submit" value="搜尋" />
</form>
```

提交後的 URL：

```
https://example.com/search?keyword=javascript
```

重要觀念：

- GET 方法的資料**永遠在 URL 中**
- `enctype` 對 GET 方法**無效**（因為明信片根本沒有「紙箱內部（Body）」，既然沒有紙箱，當然就不用貼「箱子內容物標籤」）
- 資料會出現在瀏覽器歷史記錄中

### 2.3 POST 方法的特性

```html
<!-- POST 方法的表單 -->
<form method="post" action="/submit">
    <input type="text" name="username" value="john" />
    <input type="password" name="password" value="secret123" />
    <input type="submit" value="登入" />
</form>
```

提交後的 URL：

```
https://example.com/submit
```

（注意：URL 保持乾淨，沒有參數）

重要觀念：

- 🚨 POST 方法的資料在 **HTTP Body 中**（`username=john&password=secret123`）
- URL 保持乾淨，敏感資料不會出現在 URL 中
- 因為是紙箱，這時候就**必須貼上 Content-Type**，告訴後端該怎麼拆包裹
- 只有 POST、PUT、PATCH 等方法才會有 Body

### 2.4 跟 JSON 有沒有關係？

GET / POST 本身是一種「運送方式」，跟你要寄什麼（JSON）**沒有絕對關係**。但因為 JSON 結構通常很長又有很多特殊符號（`{}`、`""`），實務上極少把它硬塞在 GET 的網址裡，所以傳送 JSON 時，99% 都是裝在 POST（紙箱）裡面傳送。

### 2.5 什麼時候該用 GET？（實務判斷力）

假設你要開發 Portal 系統的「員工搜尋」功能，希望使用者能把搜尋結果的網址複製下來，用 Teams 傳給其他同事，同事點開就能看到一樣的搜尋結果——這時候就該用 **GET**。

因為 GET 會把搜尋條件（例如 `?dept=IT&name=john`）直接寫在網址上。同事點開這串網址，就等於拿著同一張「明信片」去向伺服器要資料，看到的畫面會跟你一模一樣。相反地，如果用 POST，網址只會剩下乾淨的 `/search`，條件都藏在暗箱裡，同事複製網址打開只會看到一個空白的搜尋頁面。

> 📌 **小結**：實務上，只要是「單純查詢」、「希望可以被加進書籤」或「需要分享網址」的情境，一律優先使用 GET。GET 用網址（明信片）要資料，沒有 Body 也不需標籤；POST 用 Body（紙箱）送資料，容量大且必須貼上 Content-Type 標籤。

## 3. HTML 表單的 enctype：三種包裹的種類

### 3.1 enctype 基礎概念

**enctype** = **enc**oding **type**（編碼類型），是 HTML `<form>` 決定「包裹標籤」的設定，你可以把它想像成是在「選擇包裹的種類」。

```html
<!-- 完整語法 -->
<form method="post" action="/submit" enctype="編碼類型">
    <!-- 表單內容 -->
</form>
```

重要觀念：

- enctype **只對 POST 方法有效**
- GET 方法的資料永遠在 URL 中，不受 enctype 影響
- 在 HTML 標籤裡我們叫它 `enctype`，但當瀏覽器實際把包裹寄出去時，它在 HTTP 請求上寫的就是 `Content-Type`——**這兩個詞指的完全是同一件事**

### 3.2 第一種：application/x-www-form-urlencoded（預設的平信信封 ✉️）

#### 基本使用

```html
<!-- 這兩種寫法完全一樣 -->
<form method="post" action="/submit">
    <input type="text" name="username" value="john" />
    <input type="text" name="email" value="test@asus.com" />
    <input type="submit" value="送出" />
</form>

<!-- 明確寫出來 -->
<form method="post" action="/submit" enctype="application/x-www-form-urlencoded">
    <input type="text" name="username" value="john" />
    <input type="text" name="email" value="test@asus.com" />
    <input type="submit" value="送出" />
</form>
```

#### 實際傳送的內容

```
POST /submit HTTP/1.1
Content-Type: application/x-www-form-urlencoded
Content-Length: 34

username=john&email=test%40asus.com    ← 🚨 存在 HTTP Body 中
```

#### 為什麼資料長這樣？（背後的脈絡）

1. **用 `&` 串連**：瀏覽器把每一欄的名稱（name）和值（value）用 `=` 連起來，不同欄位之間用 `&` 隔開。這是一種最傳統、最節省空間的格式。
2. **URL 編碼（urlencoded）**：注意信箱裡的 `@` 符號變成了 `%40`。這是因為網路傳輸時，有些特殊符號會和系統原本的符號衝突，所以瀏覽器會自動進行「編碼」：
    - `@` 符號變成 `%40`
    - 空格變成 `+` 或 `%20`
    - `&` 符號變成 `%26`

#### 怎麼記憶這串長名字？

不要死背，把它切成四個部分來看：**x - www - form - urlencoded**

|片段|意義|
|---|---|
|`x-`|早期網路標準裡，`x-` 開頭代表「實驗性 (eXperimental)」或「非正式」的規格（現在大家都在用，但老名字保留了下來）|
|`www`|World Wide Web，代表這是在網頁上用的|
|`form`|表單。資料是來自網頁上的 `<form>`|
|`urlencoded`|URL 編碼。把 `@` 等符號編碼成網址看得懂的安全格式|

💡 這就像是你跟伺服器說：「這是一包**網頁(www)**來的**表單(form)**資料，而且我已經把它**翻譯成網址安全密碼(urlencoded)**了！」以後看到這個長字串，用眼睛掃過去抓到關鍵字 `form` 和 `url`，大腦就會自動連線：「喔！這就是一般純文字表單預設的格式！」

#### 後端接收（.NET Core）

```csharp
[HttpPost]
public IActionResult Submit(string username, string email)
{
    // .NET Core 會自動解碼和 Model Binding
    Console.WriteLine($"使用者: {username}");  // "john"
    Console.WriteLine($"信箱: {email}");       // "test@asus.com"（%40 自動變回 @）

    return View();
}
```

> 📌 **小結**：傳統表單送出文字資料時，瀏覽器預設會貼上 `application/x-www-form-urlencoded` 標籤，並把資料轉成安全編碼後的 `key=value&key=value` 格式送出。**千萬別用來傳檔案**。

### 3.3 第二種：multipart/form-data（檔案上傳的分隔便當盒 🍱）

#### 為什麼需要它？

如果要寄送的東西有「一張 Excel 報表（檔案）」加上「你的姓名（文字）」，這就像你要寄一個大包裹：裡面既有實體物品（檔案），又有一般文件（文字）。「平信信封」只能把文字串成 `a=1&b=2`，沒辦法把「檔案的二進位資料」塞進去。

為了解決這個問題，我們需要一個「有分隔夾層的包裹」——就像日式便當盒 🍱 一樣，飯、菜、肉要分開放，味道才不會混在一起。

一樣來拆解幫助記憶：

- **multipart**：multi（多個）+ part（部分）。代表這個包裹裡面「分成了很多區塊」
- **form-data**：表單資料

#### 基本使用

```html
<!-- Content-Type: multipart/form-data; boundary=----WebKitFormBoundary... -->
<form method="post" action="/upload" enctype="multipart/form-data">
    <input type="text" name="username" value="john" />
    <input type="file" name="uploadFile" />
    <input type="submit" value="上傳" />
</form>
```

#### 🔥 常見的踩坑經驗：忘記加 enctype

```html
<!-- ❌ 錯誤：忘記設定 enctype -->
<form method="post" action="/upload">
    <input type="file" name="uploadFile" />
    <input type="submit" value="上傳" />
</form>
```

結果：檔案上傳會失敗，因為預設的 `application/x-www-form-urlencoded` 無法處理二進位檔案。你的後端只會收到檔案的「名字」（例如字串 `report.xlsx`），但完全收不到檔案真實的二進位內容，一存檔程式就當機。

這就像是你寄信告訴對方「我要給你一罐辣椒醬」，但信封裡只有一張寫著「辣椒醬」的紙條，根本沒實體。

#### 實際傳送的內容（瀏覽器負責產生和組織格式，包含 boundary 的產生）

```
POST /upload HTTP/1.1
Content-Type: multipart/form-data; boundary=----WebKitFormBoundary7MA4YWxkTrZu0gW
Content-Length: 2048

------WebKitFormBoundary7MA4YWxkTrZu0gW              ← 開始標記
Content-Disposition: form-data; name="username"      ← 欄位資訊

john                                                 ← 文字欄位的值
------WebKitFormBoundary7MA4YWxkTrZu0gW              ← 分隔標記
Content-Disposition: form-data; name="uploadFile"; filename="report.pdf"
Content-Type: application/pdf                        ← 檔案類型

[這裡是 PDF 檔案的實際二進位內容]

------WebKitFormBoundary7MA4YWxkTrZu0gW--            ← 結束標記（注意最後有兩個 --）
```

#### Boundary（隔板）的作用

這是便當盒的靈魂。逐一拆解上面的範例：

1. **宣告隔板款式**：第一行的 `Content-Type` 旁邊，瀏覽器自動加了一串亂碼 `boundary=----WebKit...`。這就像在包裹外面註明：「嗨！我這次便當盒用的塑膠隔板款式是這串代碼喔！」
2. **區隔每道菜**：包裹內部，每一筆資料（每一道菜）之間，都塞了一塊一模一樣的隔板。
3. **便當盒的底部**：最後一行的隔板，尾巴多了兩個減號 `--`，專門用來告訴後端：「便當盒到底了，後面沒有菜了。」

重點整理：

- `boundary` 是**瀏覽器自動產生的唯一字串**（一串不會跟真實資料撞名的亂碼）
- 用來把 HTTP Body 切割成好幾個獨立的夾層，讓文字和檔案可以安全地放在同一個包裹裡傳送
- 如果沒有隔板，會流動的滷肉汁（圖片檔案）和乾的白飯（純文字）會全部混在一起，後端完全分不清「哪裡是文字的結束、哪裡是圖片的開始」，讀出來的檔案就會損毀

#### 後端接收（.NET Core）

```csharp
[HttpPost]
public IActionResult Upload(string username, IFormFile uploadFile)
{
    // .NET Core 自動處理 multipart/form-data 的解析
    // 你不需要手動處理 boundary

    Console.WriteLine($"使用者: {username}");

    if (uploadFile != null && uploadFile.Length > 0)
    {
        Console.WriteLine($"檔案名稱: {uploadFile.FileName}");
        Console.WriteLine($"檔案大小: {uploadFile.Length} bytes");
        Console.WriteLine($"檔案類型: {uploadFile.ContentType}");

        // 儲存檔案
        var filePath = Path.Combine("uploads", uploadFile.FileName);
        using (var stream = new FileStream(filePath, FileMode.Create))
        {
            uploadFile.CopyTo(stream);
        }
    }

    return View();
}
```

> 📌 **小結**：純文字就交給預設的 urlencoded；**只要表單裡有檔案，就必須換成「多夾層便當盒」multipart/form-data**。如果只有純文字，反而不建議用便當盒（包裝太厚重，浪費網路速度）。

### 3.4 第三種：text/plain（很少使用）

#### 基本使用

```html
<form method="post" action="/submit" enctype="text/plain">
    <input type="text" name="username" value="john" />
    <input type="text" name="email" value="test@asus.com" />
    <input type="submit" value="送出" />
</form>
```

#### 實際傳送的內容

```
POST /submit HTTP/1.1
Content-Type: text/plain
Content-Length: 35

username=john
email=test@asus.com
```

#### 與 application/x-www-form-urlencoded 的比較

|比較項目|text/plain|x-www-form-urlencoded|
|---|---|---|
|資料長相|`username=john`（換行分隔）|`username=john&email=test%40asus.com`|
|分隔符號|`\n`（換行）|`&`|
|編碼處理|不編碼|URL 編碼|
|解析難度|需要手動解析|框架自動解析|

#### 為什麼很少使用？

1. **沒有標準**：沒有統一的解析規範
2. **處理困難**：需要手動解析，容易出錯
3. **不支援檔案**：無法處理二進位資料
4. **相容性差**：不是所有後端框架都支援

## 4. 容易混淆的標籤比較

### 4.1 x-www-form-urlencoded vs text/html：看起來都是純文字，差在哪？

- **x-www-form-urlencoded（平信）**：傳送的是「**結構化的鍵值對**」。就像填寫問卷，有特定的欄位與對應的值（例如 `name=john&age=18`）。
- **text/html**：傳送的是「**一整塊網頁排版程式碼**」。就像你直接把一整頁報紙的排版內容（例如 `<div><h1>哈囉</h1></div>`）當作包裹寄過去。

**$.ajax() 的 contentType 能設定成 text/html 嗎？**

可以，但實務上「發送」時極少使用：

- **發送（Request）**：極少。只有在特殊情境，例如你在開發一個像 Word 一樣的網頁文章編輯器，需要把使用者寫好的整篇 HTML 原始碼存進資料庫時，才會這樣設定。
- **回傳（Response）**：非常常見。這通常是伺服器回傳給瀏覽器時使用的標籤，告訴瀏覽器：「這是一張網頁，請幫我畫出畫面。」

🚨 **踩坑經驗**：如果你真的用 AJAX 傳送 HTML 給 .NET 後端，常常會觸發後端的「資安防護機制（Request Validation）」，後端會以為這是一次駭客的 XSS 惡意攻擊而直接阻擋報錯。

### 4.2 瀏覽器怎麼「看」標籤？（絕對盲從的郵差）

瀏覽器其實很死板：當它從伺服器收到包裹時，**不會去偷看裡面的內容長怎樣，只會死死盯著外箱上的 Content-Type 標籤**來決定接下來的動作。

假設伺服器回傳的文字一模一樣，都是 `<h1>你好，我是標題</h1>`：

|情境|標籤|瀏覽器的反應|使用者看到的畫面|
|---|---|---|---|
|A|`text/html`|「這是網頁設計圖，我要啟動畫筆！」|`<h1>` 被藏起來，「你好，我是標題」變成又大又粗的黑體字|
|B|`text/plain`（或平信）|「這是純文字，照抄就對了。」|原封不動顯示 `<h1>你好，我是標題</h1>` 這整串原始碼文字|

💡 這就像在寶特瓶裡裝白開水：貼上「飲用水」標籤，別人拿到就直接喝下去（解析並顯示）；貼上「有毒溶液」標籤，別人就只會小心翼翼放在桌上觀察，絕對不會喝它（不解析，直接把原始碼顯示給你看）。

> 📌 **小結**：瀏覽器不會管你裡面裝了什麼，它完全聽命於 Content-Type 標籤。貼什麼標籤，它就啟動什麼對應的顯示模式。這也是為什麼後端回傳時，絕對不能貼錯標籤。

## 5. 常用 Content-Type 完整指南

### 5.1 主要用於請求（Request）的標籤

|Content-Type|何時使用|資料格式|
|---|---|---|
|`application/x-www-form-urlencoded`|HTML 表單提交（預設）|`key=value&key=value`，會進行 URL 編碼|
|`multipart/form-data`|檔案上傳或混合資料|使用 boundary 分隔；必須設定 `enctype="multipart/form-data"`|
|`application/json`|API 呼叫、結構化資料傳送（AJAX 請求、REST API）|JSON 字串|

### 5.2 主要用於回應（Response）的標籤

|Content-Type|何時使用|瀏覽器行為|
|---|---|---|
|`text/html`|網頁內容|解析 HTML 標籤，建立 DOM，渲染頁面|
|`text/css`|CSS 樣式表|解析 CSS 規則，套用到元素|
|`application/javascript`|JavaScript 檔案|解析並執行 JavaScript 程式碼|
|`application/json`|API 回應、AJAX 資料|jQuery 自動解析成 JavaScript 物件|
|`application/pdf`|PDF 檔案|嘗試顯示 PDF 或提示下載|
|`application/octet-stream`|檔案下載|🚨 強制彈出「另存新檔」視窗（不管什麼檔案都適用）|
|`image/png`、`image/jpeg`|圖片檔案|顯示圖片|

### 5.3 Excel 的完整 Content-Type

現代的 `.xlsx` 格式，完整標籤非常長：

```
application/vnd.openxmlformats-officedocument.spreadsheetml.sheet
```

拆解幫助記憶：

- **vnd**：代表 Vendor（供應商），表示這是特定廠商（微軟）制定的格式
- **openxmlformats**：代表這是基於 XML 的開放格式（這就是為什麼副檔名最後有一個 x，變成 `.xlsx`）
- **spreadsheetml.sheet**：試算表（Spreadsheet）工作表

實務上這串通常不需要硬背，需要用時直接搜尋「Excel content type」複製貼上即可。只要認得 `spreadsheet` 就能分辨。

```csharp
// Excel 檔案 (.xlsx)
return File(excelBytes,
    "application/vnd.openxmlformats-officedocument.spreadsheetml.sheet",
    "data.xlsx");
```

```csharp
// 🚨 強制下載任何檔案
return File(fileBytes, "application/octet-stream", "file.zip");
```

### 5.4 Content-Type 總對應表

|Content-Type|主要用於|說明|
|---|---|---|
|`application/x-www-form-urlencoded`|**請求**|HTML 表單預設格式|
|`multipart/form-data`|**請求**|檔案上傳必須用|
|`application/json`|**兩者**|API 資料交換|
|`text/plain`|**回應**|顯示純文字|
|`text/html`|**回應**|網頁內容|
|`text/css`|**回應**|樣式表檔案|
|`application/javascript`|**回應**|JavaScript 檔案|
|`application/pdf`|**回應**|PDF 檔案|
|`application/octet-stream`|**回應**|檔案下載|
|`image/png`、`image/jpeg`|**回應**|圖片檔案|

### 5.5 你的開發中，使用頻率排序

1. **application/json** — API 開發最常用
2. **application/x-www-form-urlencoded** — 表單提交
3. **multipart/form-data** — 檔案上傳
4. **text/html** — 網頁回應
5. **application/vnd.openxmlformats-officedocument.spreadsheetml.sheet** — Excel 下載
6. **application/octet-stream** — 檔案下載

## 本篇總結

1. **Content-Type 是標籤**：告訴接收方資料格式；請求與回應兩個方向都可能有；大部分情況瀏覽器自動設定；錯誤的 Content-Type 會導致解析失敗。
2. **GET vs POST**：GET 是明信片（資料永遠在 URL，沒有 Body，不受 enctype 影響）；POST 是密封紙箱（資料在 HTTP Body，必須貼標籤）。
3. **三種表單 enctype**：
    - `application/x-www-form-urlencoded`：預設，一般表單，格式為 `key=value&key=value`
    - `multipart/form-data`：檔案上傳必須，使用 boundary 分隔
    - `text/plain`：很少使用，格式為 `key=value\nkey=value`
4. **enctype 就是 Content-Type**：HTML 用的名字叫 enctype，寄出去時寫在 HTTP 上的名字叫 Content-Type，同一件事。
