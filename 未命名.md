## 1. 為什麼要學這個？

在前端開發中，你很可能已經遇過（或者未來一定會遇到）這些令人抓狂的情境：

- 「我明明用程式改了輸入框的值，畫面上卻沒有變化！」
- 「我去抓 Checkbox 的勾選狀態，怎麼感覺怪怪的？」
- 「表單 reset 之後，明明清空過的欄位，值又跑回來了！」

這些 Bug 的根源幾乎都指向同一個問題：**搞混了 HTML Attribute（屬性）和 DOM Property（狀態）。**

學懂這兩者的差異，能幫你釐清瀏覽器的運作邏輯，讓你在精準操作表單元素時不再靠猜——這是前端開發的基礎內功，早學早受益。

---

## 2. 最核心的比喻：出生證明 vs 當前狀態

在我們看任何程式碼之前，先把最重要的觀念印在腦子裡。

當瀏覽器讀取一段 HTML 的時候，它其實做了兩件事：

**第一步**：解析 HTML 標籤，把上面寫的資料當作「初始設定」記下來。 **第二步**：在記憶體裡建立一個活生生的 DOM 物件（一個 JavaScript 可以操作的物件）。

這兩個東西雖然出身相同，但從那一刻起，就開始過著各自的生命：

|概念|比喻|代表意義|
|:--|:--|:--|
|**HTML Attribute（屬性）** 🪪|出生證明|元素的「初始預設值」，是與生俱來的，記錄「一開始是什麼樣子」|
|**DOM Property（狀態）** 🏃‍♂️|當前狀態|元素「現在的真實狀態」，會隨使用者的操作隨時改變|

**核心小結：Attribute 負責記錄「初始預設值」，Property 負責反映「當前即時狀態」。** 這句話請先背起來，後面所有的觀念都從這裡展開。

## 3. 基礎對決：attr vs prop vs val

### 3.1 一張表搞懂差異

現在我們來把這個比喻對應到實際的程式碼語法：

|特性|HTML Attribute（出生證明）🪪|DOM Property（當前狀態）🏃‍♂️|
|:--|:--|:--|
|**資料來源**|寫在 HTML 標籤上的內容|瀏覽器在記憶體中建立的活物件|
|**資料型態**|永遠是「字串 (String)」|可能是字串、布林值（true/false）等|
|**代表意義**|元素的「初始預設值」|元素的「當前即時狀態」|
|**原生 JS 寫法**|`element.getAttribute('value')`|`element.value`|
|**jQuery 寫法**|`.attr('value')`|`.prop('value')` 或 `.val()`|

### 3.2 用一個例子把差異說清楚

假設網頁上有這樣一個輸入框：

```html
<input type="text" id="fruit" value="蘋果">
```

網頁載入後，使用者在畫面上把「蘋果」刪掉，手動輸入了「香蕉」。

這時候，`attr` 和 `prop` 會給你完全不同的答案：

```javascript
// 使用者輸入「香蕉」之後...

$('#fruit').attr('value')
// → "蘋果"（HTML 標籤上寫的初始值，永遠不受使用者操作影響）

$('#fruit').prop('value')
// → "香蕉"（使用者當下輸入的，代表目前的真實狀態）

$('#fruit').val()
// → "香蕉"（.val() 是 jQuery 針對表單元素的專用語法，等同於 .prop('value')）
```

> ⚠️ **實務建議**：在讀取或修改輸入框的值時，因為我們處理的是「當前狀態」，請優先使用 `.val()` 或 `.prop('value')`，而不是 `.attr('value')`。`.attr()` 讀到的是「初始值」，不是使用者現在輸入的內容，這是新手最常踩的坑之一。

## 4. 同步機制：誰會跟著改變、誰不會？

理解了 Attribute 和 Property 的差異之後，下一個問題自然就來了：**當 Property 改變的時候，Attribute 會跟著改嗎？**

答案是：**要看屬性的「性質」，瀏覽器有不同的處理策略。**

### 4.1 表單資料類（value、checked）——不同步，刻意設計的

拿「手機鈴聲」來比喻：手機出廠時預設鈴聲是嗶嗶聲（Attribute）。當你手動把鈴聲換成流行歌（Property），這個改變**不會**覆蓋掉出廠設定。為什麼？因為如果同步覆蓋了，你就永遠無法「恢復原廠設定」了！

在網頁表單中也是一樣的邏輯。當使用者在 `<input>` 裡打了新的字（改變了 Property），瀏覽器刻意不把新字寫回 HTML Attribute，為的就是「保留初始預設值」，確保 `form.reset()` 被呼叫時，瀏覽器知道要把欄位還原回什麼狀態。

### 4.2 身分結構類（id、class）——自動同步，因為必須一致

`id` 和 `class` 就像是元素的名字和職稱。如果你用 JavaScript 改了元素的 class（Property），HTML 標籤上的 class（Attribute）**必須**跟著自動更新。

為什麼？因為 CSS 樣式規則和 JavaScript 的 DOM 選擇器，都是依賴 HTML 標籤上的 class 和 id 來運作的。如果這兩者不同步，CSS 會對應不到，畫面排版會直接壞掉。

### 4.3 整理成表格

|屬性類別|同步行為|背後原因|常見元素|
|:--|:--|:--|:--|
|表單資料類（`value`、`checked`）|❌ **不同步**|必須保留出廠預設值，確保 `form.reset()` 能正確還原|`<input>`, `<checkbox>`|
|身分結構類（`id`、`class`）|✅ **自動同步**|必須維持 HTML 與 DOM 一致，否則 CSS 與 JS 選擇器會失效|所有 HTML 元素|

### 4.4 一個很容易踩的坑：CSS 屬性選擇器

這個同步機制和 CSS 的搭配有一個很容易踩的坑，請特別注意。

假設你想讓沒有填寫值的欄位自動變成紅框，你可能會這樣寫 CSS：

```css
/* ⚠️ 這個寫法有問題！ */
input[value=""] {
    border: 1px solid red;
}
```

你以為當使用者清空輸入框時，框框就會變紅。但**不會**！因為使用者清空的是 Property，而 CSS 的 `[value=""]` 監聽的是 Attribute，而 Attribute 並不會跟著同步清空。

**正確的做法**是透過 jQuery 偵測輸入內容，在適當時機用 `.addClass()` 加上一個 class，透過會自動同步的 class 來控制 CSS——這個主題我們在第 8 章會完整說明。

## 5. 防禦性清空：val 搭配 attr 的實務場景

學完同步機制，我們來解析一個在實務程式碼中偶爾會看到、讓人覺得「這為什麼要這樣寫」的奇特寫法：

```javascript
$element.val('').attr('value', '');
```

### 5.1 為什麼 .val('') 還不夠？

`.val('')` 確實已經把畫面上的文字清空了（修改了 Property）。那為什麼還要額外去清空 Attribute？

**答案是：為了防止 `form.reset()` 的「舊值回魂」。**

當程式呼叫 `form.reset()` 時，瀏覽器的邏輯是：「把畫面上所有元件，都還原回 HTML Attribute 上的初始狀態」。如果我們只清空了 Property，Attribute 裡還留著舊資料，一執行 reset，舊資料就會像幽靈一樣回到畫面上。

### 5.2 什麼時候該用、什麼時候不用

所以這兩種寫法各有適用情境：

**一般清空（只改 Property）**：如果你的清除按鈕只是單純讓畫面變空白，後續流程根本不會用到 `form.reset()`，那寫 `$element.val('')` 就非常足夠了。

```javascript
// 單純清空畫面，這樣就夠了
$('#searchInput').val('');
```

**嚴謹清空（Property + Attribute 同步）**：如果你的系統流程是「清除畫面 ➡ 之後可能會觸發 reset」，為了防止舊值回魂，才需要加上 `.attr('value', '')`。

```javascript
// 防禦性清空，同時抹除初始值，避免 reset 時舊資料回魂
$('#searchInput').val('').attr('value', '');
```

> 💣 **踩坑紀錄：過度防禦** 有些工程師知道這個坑之後，會乾脆寫 `$element.removeAttr('value')` 把整個屬性連根拔起。其實這沒必要，除非你的 CSS 剛好有寫 `input[value=""] { ... }` 這種針對屬性選擇器的樣式，否則只要把屬性值設成空字串就已經足夠安全了。在 Portal 或 GTS 系統中，清除表單大多是配合 Modal（彈出視窗）開關，通常不需要拔除屬性。

**本章小結**：只清畫面用 `.val()` 即可；若要防止表單重置回魂，才需要加 `.attr('value', '')` 來同步清空初始值。

## 6. 表單實戰：Checkbox 與 Select 的正確操作

文字輸入框的情境比較單純，但 Checkbox（核取方塊）和 Select（下拉選單）才是最常讓人搞混的重災區。

### 6.1 Checkbox——布林值狀態請用 .prop()

**需求情境**：使用者點擊按鈕，你要用程式強制幫他把「同意服務條款」的 Checkbox 打勾。

```javascript
// ❌ 錯誤寫法（不要這樣做）
$('#terms').attr('checked', true);
// 這樣寫在舊版 jQuery 或特定情境下，畫面上的框框根本不會被打勾！
// 你只是在背後的 HTML 標籤上加了字，沒有真正觸發元件狀態的改變。

// ✅ 正確寫法
$('#terms').prop('checked', true);
// 這才是修改「當前狀態」，畫面會立即反應。
```

**為什麼？** 把 Checkbox 打勾是一個「當下狀態」的改變，不是修改「出生證明」。同樣的道理，要取消勾選就用 `.prop('checked', false)`，要讀取目前是否打勾就用 `.prop('checked')`（會回傳 `true` 或 `false`）。

> ⚠️ **通用原則**：處理布林值（true/false）狀態如 `checked`、`disabled`、`readonly`，一律使用 `.prop()` 就對了。

### 6.2 Select 下拉選單——.val() 是萬用武器

處理下拉選單的取值和給值是很頻繁的操作。很多人會直覺想說：是不是要去找出那個 `<option>`，然後對它設定 `prop('selected', true)`？

其實 jQuery 提供了一個針對所有表單元素的萬用武器：`.val()`。

```javascript
// 取得目前選中的值
$('#citySelect').val();
// → 回傳當前選中的值，例如："taipei"

// 用程式動態改變選項（選到台南）
$('#citySelect').val('tainan');
// jQuery 會自動幫你找到對應的 <option value="tainan">，並把它切換過去。
```

雖然你「可以」用 `$('#citySelect option:first').prop('selected', true)` 來切換選項，但在實務上這樣寫程式碼長且難以閱讀。只要是處理表單元素的具體數值，強烈建議直接使用 `.val()`，它會自動搞定背後的狀態同步。

**本章小結**：處理表單的當前動態狀態（勾選、啟用、停用）請找 `.prop()`；處理表單的具體內容數值（輸入框的字、下拉選單的選項）請找 `.val()`。

## 7. 便利貼資料：data-* 屬性與 .data() 方法

### 7.1 什麼是 data-* 屬性？

HTML5 給了工程師一個很實用的功能：**自訂資料屬性（Data Attributes）**。你可以把它想像成是在元素上貼「便利貼」📝。

以前如果我們想在一個 `<div>` 上偷藏使用者的 ID 或身分，沒有標準的地方可以放；現在只要用 `data-` 開頭，你想貼什麼都可以：

```html
<div id="user" data-id="123" data-role="admin"></div>
```

### 7.2 讀取便利貼：.attr() vs .data()

讀取便利貼有兩種方式，它們的結果並不完全一樣：

```javascript
// 方法一：用 .attr()（你第一直覺想到的方法）
$('#user').attr('data-id');
// → 字串 "123"（忠實呈現 HTML 上的字串，包含 "data-" 前綴）

// 方法二：用 .data()（推薦寫法）
$('#user').data('id');
// → 數字 123（jQuery 會自動轉型，而且不需要寫 "data-" 前綴）
```

兩種方法都能抓到值，但 `.data()` 更聰明：

- **語法更簡潔**：不需要寫 `data-` 前綴，直接寫你的 key 就好。
- **自動型別轉換**：如果便利貼上寫的是數字 `"123"`，用 `.attr()` 抓出來是字串 `"123"`，要做加減乘除還要自己轉換；用 `.data()` 抓出來就直接是數字 `123`，可以馬上使用。

> ⚠️ **實務建議**：處理 `data-*` 屬性時，雖然 `.attr()` 能用，但強烈建議改用 `.data()`，語法更簡潔，還能幫你自動處理資料型態。

### 7.3 修改便利貼：超級大坑請注意！

這裡有一個無數新手甚至老手都會踩的超級大坑。如果你用 `.data()` 去**修改**便利貼的內容：

```javascript
$('#user').data('role', 'guest');
```

**這個修改只會存在於 jQuery 的記憶體中，它不會去更改 HTML 標籤上的字！**

你這時候打開瀏覽器的開發者工具去檢查 HTML，你會看到畫面上依然寫著 `data-role="admin"`。`.data()` 的修改就像是在你腦海中換了一個便利貼，但牆上貼的那張（HTML）根本沒動到。

```javascript
// 執行修改（只改了記憶體）
$('#user').data('role', 'guest');

// 馬上用 .attr() 去讀 HTML 標籤上的值
$('#user').attr('data-role');
// → "admin"（因為 HTML 根本沒被改到，抓到的還是原本的值）
```

> 💣 **踩坑場景**：如果你的 CSS 有寫 `div[data-role="guest"] { color: red; }` 這種綁定在 HTML 屬性上的樣式，那你用 `.data()` 去改值，畫面是不會變紅的，因為 HTML 根本沒變。這時你必須改用 `.attr('data-role', 'guest')` 才能真正修改 HTML 標籤上的屬性。

**本章小結**：

- 讀取 `data-*` 屬性：優先使用 `.data()`，享受自動型別轉換的好處。
- 修改 `data-*` 屬性且**不需要**同步反映到 HTML 標籤：用 `.data()` 修改即可。
- 修改 `data-*` 屬性且**需要**同步反映到 HTML 標籤（例如配合 CSS 屬性選擇器）：改用 `.attr()` 修改。

## 8. 樣式操作：為什麼不要用 attr 改 class

### 8.1 兩種改變外觀的方式

在程式裡改變元素的外觀，主要有兩種層次：

**直接修改 style（行內樣式）——像是「刺青」**：把顏色、大小直接寫死在元素上（例如 `style="color: red;"`）。見效快，但系統大了之後非常難統一修改和管理。

**切換 Class（類別）——像是「換衣服」**：事先在 CSS 檔案裡把樣式設計好（例如設計一套叫做 `.error-state` 的紅框樣式），然後在程式碼中視情況讓 HTML 元素穿上或脫下這套衣服。

> ⚠️ **實務建議**：避免用 JavaScript 直接寫死 style（刺青），請盡量透過切換 class（換衣服）來改變外觀，讓「JavaScript 負責邏輯、CSS 負責美觀」，各司其職，未來維護才會輕鬆。

### 8.2 為什麼不能用 .attr('class') 來換衣服

既然要換衣服，直覺可能會想到用之前學過的 `.attr()` 來處理 class。但這樣做有個致命問題。

假設你的 HTML 一開始長這樣，這個 `<div>` 本來就穿著一件負責排版的衣服 `container`：

```html
<div id="myBox" class="container"></div>
```

現在你想讓它多穿一件叫做 `active` 的外套：

```javascript
// ❌ 錯誤寫法
$('#myBox').attr('class', 'active');

// 結果：<div id="myBox" class="active"></div>
// container 不見了！.attr() 是直接覆寫整個 class 屬性，
// 把原本穿的衣服（排版用的 container）硬生生扒光了。
```

### 8.3 正確的換衣服方式

jQuery 提供了一整套專門用來「管理衣櫃」的方法，讓你可以在不影響其他樣式的情況下自由穿脫：

```javascript
// ✅ 穿上新衣服（只加 active，container 安全地留著）
$('#myBox').addClass('active');
// 結果：class="container active"

// ✅ 脫掉指定衣服（精準拿掉，其他樣式不受影響）
$('#myBox').removeClass('active');
// 結果：class="container"

// ✅ 智慧切換（有就脫、沒有就穿，做開關非常好用）
$('#myBox').toggleClass('visible');
```

**`toggleClass` 的最佳使用情境**——漢堡選單：

```javascript
// 使用者每次點擊漢堡按鈕時，只需要一行程式碼控制選單展開 / 收合
$('#hamburger-btn').on('click', function () {
    $('#mobile-menu').toggleClass('show');
    // 不需要 if...else 判斷現在是開還是關，jQuery 自動幫你搞定！
});
```

> 💡 **設計哲學**：處理外觀變化的最佳默契是「JavaScript 負責切換狀態標籤（Class）🎛️，CSS 負責定義真正的視覺外觀 🎨」。各司其職，不要用 JS 去寫死顏色或大小，未來的維護才會輕鬆。

**本章小結**：絕對避免用 `.attr('class')` 暴力覆寫樣式，請一律透過 `.addClass()`、`.removeClass()`、`.toggleClass()` 來精準穿脫衣服。

## 9. 完整速查表

學完整份文件，把所有語法整理在一張表，方便日後查閱：

### 9.1 核心觀念速查

|我想處理的是...|用什麼語法|原因|
|:--|:--|:--|
|讀取 HTML 標籤上的初始值|`.attr('屬性名')`|這是 Attribute（出生證明）|
|修改 HTML 標籤上的屬性|`.attr('屬性名', '新值')`|同上|
|讀取或修改輸入框的當前內容|`.val()` 或 `.val('新值')`|這是表單元素的當前狀態|
|讀取或修改布林值狀態（checked、disabled）|`.prop('屬性名')` 或 `.prop('屬性名', true/false)`|布林值是 Property，不是 Attribute|
|讀取 data-* 自訂屬性|`.data('key')`|自動型別轉換，語法簡潔|
|修改 data-* 自訂屬性（不需要影響 CSS 選擇器）|`.data('key', '新值')`|只改記憶體|
|修改 data-* 自訂屬性（需要影響 CSS 屬性選擇器）|`.attr('data-key', '新值')`|需要改 HTML 標籤本身|
|新增樣式 class|`.addClass('class名')`|只新增，不覆蓋其他 class|
|移除樣式 class|`.removeClass('class名')`|精準移除|
|切換樣式 class（開關用）|`.toggleClass('class名')`|有就拿掉，沒有就加上|

### 9.2 清空表單欄位的選擇

|情境|推薦寫法|
|:--|:--|
|單純清空畫面（不會觸發 form.reset()）|`$element.val('')`|
|嚴謹清空（之後可能觸發 form.reset()）|`$element.val('').attr('value', '')`|
|需要連根清除 Attribute（幾乎不需要）|`$element.removeAttr('value')`（非必要，謹慎使用）|

### 9.3 大原則一句話總結

> **記住這個優先順序**： 讀寫表單值 → `.val()` 讀寫布林狀態 → `.prop()` 讀取 data 屬性 → `.data()` 讀取初始值或操作 HTML 標籤本身 → `.attr()` 切換樣式外觀 → `addClass / removeClass / toggleClass`（**絕對不要用 `.attr('class')`**）

## 附錄：延伸閱讀

如果你想繼續深入這些概念，有幾個方向可以探索：

**原生 JavaScript 的 dataset API**：搜尋「MDN HTMLElement.dataset」，這是瀏覽器原生處理 `data-*` 便利貼的 API，概念跟 jQuery 的 `.data()` 讀取方式幾乎一樣，但修改行為略有差異——`dataset` 的修改會直接反映到 HTML 標籤上，這點與 jQuery `.data()` 不同。

**jQuery 版本差異**：你的 Portal 系統使用 jQuery 3.3.1，GTS 系統使用 jQuery 1.10.2。在舊版 jQuery 中，`.attr()` 和 `.prop()` 的邊界曾經較為模糊，某些布林屬性的行為與新版不同。若在 GTS 系統遇到怪異的 checked 行為，這可能是版本差異造成的。
