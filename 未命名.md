## 寫在前面：給剛展開資料表的你

嘿，看到你開始好奇 SSMS 資料表底下那些「資料夾」是什麼，我真的很開心。這代表你已經從「只是把資料塞進去」的階段，準備往「懂得設計與優化資料庫」的方向前進了。

這五個節點——**主鍵、索引鍵、條件約束、索引、統計資料**——剛看到的時候真的容易眼花撩亂，因為它們的名字長得有點像，功能卻完全不同。別擔心，我帶你一個一個拆解，看完你就會發現其實沒那麼複雜。

---
## 一、最重要的心智模型：這五個節點只在做兩件事

在進入細節之前，我希望你先把這個觀念刻進腦袋裡：

> **這五個節點背後，其實只在做兩件事：「限制資料」和「加速查詢」。**

**限制資料**，是要求資料必須符合某些規則，例如員工編號不能重複、年齡不能是負數、訂單的客戶必須真實存在。這類機制統稱為「**條件約束（Constraints）**」。

**加速查詢**，是讓 SQL Server 能更快找到你要的資料。這類機制統稱為「**索引（Index）**」。

你可能會想：「那為什麼主鍵、索引鍵、條件約束、索引看起來這麼像，又被分到不同節點？」

答案是——**有些機制同時兼具「限制」和「加速」兩個身份**，所以 SSMS 為了方便管理，從不同角度各自列出來。換句話說，**同一個東西可能會在好幾個節點同時出現**。這一點非常重要，我們等一下會一直回頭呼應，你先記住就好。

我用圖書館當比喻來貫穿全文：資料表就是圖書館裡的書架。如果書亂塞一通，找書的人會崩潰、借還書也天下大亂。這五個節點，就是讓圖書館「維持秩序、確保資料正確、又能快速找到書」的五大法寶。

---
## 二、主鍵（Primary Key）——資料的身分證

### 它在解決什麼問題？

資料庫裡可能有上百萬筆資料，我們要解決的問題是：**「如何精準定位到唯一的那一筆？」**

### 用生活比喻理解

主鍵就像你的**身分證字號**。世界上可能有很多叫「陳冠宇」的人，但身分證字號只會對應到唯一的一個你。在資料表中，主鍵確保每一筆紀錄都是獨一無二的。

如果用你熟悉的 C# 來類比，主鍵的角色很像 `Dictionary<TKey, TValue>` 裡的 `TKey`——每筆資料都要有獨一無二的 key，而且這個 key 不能是 `null`。

### 主鍵的兩條鐵則

SQL Server 對主鍵會強制執行兩條規則：**欄位值不能重複**（唯一性），以及**欄位值不能為 NULL**（必填）。只要違反其中一條，資料就進不去。

### 複合主鍵：一個主鍵，多個欄位

很多新人會搞混這件事，我講清楚：

> **一張表只能有「一個」主鍵，但這個主鍵可以由多個欄位組合而成，這叫做「複合主鍵（Composite Primary Key）」。**

舉例來說，「選課表」可能用 `(學生編號, 課程編號)` 一起當主鍵——兩個欄位合起來才是一個主鍵，不是兩個主鍵。這個概念在第 3 篇的 Q1 會再深入說明。

### ⚠️ 踩坑警告：千萬不要拿會變的東西當主鍵

新手最常犯的錯，是拿「手機號碼」或「Email」這種看起來唯一的欄位當主鍵。**千萬不要！** 因為使用者的手機會換、Email 也會改。一旦變動，所有跟他有關聯的訂單、留言紀錄就會瞬間找不到主人。

**建議做法**：永遠用**無意義的流水號（Identity）**或 **UUID（全域唯一識別碼）**來當主鍵，它們一輩子不會變。

### 主鍵的一個重要隱藏行為

> **當你建立主鍵時，SQL Server 會「自動」幫你建立一個唯一索引**（預設是叢集索引）。

這就是為什麼你等一下會在「索引」節點底下也看到主鍵的身影——它同時是「條件約束」也是「索引」。這就是「同一個東西出現在多個節點」的第一個例子，記住這點，後面講「索引」節點時會再呼應。

> **⚠️ 進階補充**：這裡說的「唯一索引（Unique Index）」是底層的實體資料結構，它和「唯一鍵（Unique Key）」雖然名字像，但其實是**不同層級**的概念。我們在本篇第八節「深入辨析」會詳細解釋。

## 三、索引鍵（Keys）——所有鍵類型規則的家

這個節點容易讓人混淆，因為它叫「索引鍵」，但裡面其實放的是「**鍵值類型的條件約束**」，主要有三種：主鍵（PK）、唯一鍵（Unique Key）、外部索引鍵（Foreign Key）。主鍵剛才講過了，我們重點看後兩個。

### 唯一鍵（Unique Key）

跟主鍵很像，也是要求欄位值不能重複，但有兩個關鍵差異：

|比較項目|主鍵（PK）|唯一鍵（UK）|
|---|---|---|
|是否允許 NULL|❌ 不允許|✅ 允許（但通常只允許一個 NULL）|
|一張表的數量|只能有 1 個|可以有很多個|

舉例：員工表的主鍵是 `EmployeeId`，但你還希望 `Email` 和 `身分證字號` 都不能重複，這時就把它們設成唯一鍵。

#### 「通常只允許一個 NULL」是什麼意思？

這個 SQL Server 的特殊行為，新手第一次看到都會一頭霧水，我把它講清楚。

假設 `Employees` 表的 `Email` 欄位設了 UK，現在有兩筆資料 Email 都還沒填：

```sql
INSERT INTO Employees (Name, Email) VALUES ('小明', NULL);  -- ✅ OK
INSERT INTO Employees (Name, Email) VALUES ('小華', NULL);  -- ❌ 報錯！
```

第二筆會被擋下來。**關鍵觀念**：**允許 NULL ≠ 允許很多個 NULL**。SQL Server 在處理唯一鍵時採取一種特殊邏輯——它把 NULL 當成「一個具體的值」來看待，所以「**兩個 NULL 算重複**」。這跟一般 SQL 的 NULL 比較邏輯不太一樣（一般來說 `NULL = NULL` 會得到 `UNKNOWN`），但在唯一鍵的脈絡下，SQL Server 會把多個 NULL 視為違反唯一性。

#### ⚠️ 踩坑警告：NULL 很多的欄位不要設唯一鍵

如果你的欄位**可能會有很多筆 NULL**（例如「離職日期」，大部分員工都還在職所以都是 NULL），那**絕對不要在這個欄位上設唯一鍵**，否則第二個還沒離職的員工就進不來了。

#### 進階技巧：篩選索引（Filtered Index）

如果你真的需要「允許多個 NULL，但非 NULL 值要唯一」，SQL Server 有個超實用的解法——**篩選索引**：

```sql
-- 只對非 NULL 值檢查唯一性，NULL 愛幾個就幾個
CREATE UNIQUE INDEX UX_Employees_Email
ON Employees(Email)
WHERE Email IS NOT NULL;
```

這是實務上超常用的技巧，記下來未來會救你的命。

### 外部索引鍵（Foreign Key, FK）

這是個完全不同層次的概念，它處理的是**「表與表之間的關聯」**。

用你的工作情境舉例：假設 Portal 系統有 `Orders` 表和 `Customers` 表，把 `Orders.CustomerId` 設為外部索引鍵指向 `Customers.CustomerId`，那 SQL Server 就會幫你把關——**你不能在 `Orders` 裡插入一筆 `CustomerId` 不存在於 `Customers` 表的訂單**。這就是所謂的「參考完整性」。

用 C# 來類比，外部索引鍵就像參考類型——你不能讓一個物件指向不存在的記憶體位址。

> **小結**：「索引鍵」這個節點，你可以理解成「**所有跟 Key 有關的規則**」的集合處，包含 PK、UK、FK 三大類。

## 四、條件約束（Constraints）——資料庫的嚴格守門員

### 它在解決什麼問題？

解決「**Garbage In, Garbage Out**」的問題。如果有人把年齡填成 `-5` 歲，或把應該填數字的地方塞了中文字，下游系統就會崩潰。條件約束就是用來擋下這些髒資料的。

就像**夜店門口的保鑣**——你說你 18 歲？保鑣會檢查你的證件，不符合規定就直接踢出去，不准進入資料庫。

### SSMS 的「條件約束」節點裡放什麼？

從 **SSMS 介面的角度**來看，這個節點主要放兩種：

#### 1. 檢查約束（Check Constraint）

你對欄位值加的自訂規則，就像 C# 裡的 `if` 判斷，只是在資料庫層級執行。不管哪個系統、哪個工程師寫的程式去 INSERT 資料，都不可能塞入違反規則的值。例如：

```sql
Age >= 0 AND Age <= 150
Salary > 0
Status IN ('A', 'B', 'C')
```

#### 2. 預設值約束（Default Constraint）

當你 INSERT 一筆資料而沒有指定某個欄位值時，SQL Server 會自動幫你填入預設值。例如：

```sql
CreatedDate DEFAULT GETDATE()
```

### 進階觀念：其實 PK、UK、FK 也都是「條件約束」

你如果繼續追問：「PK、UK 是不是也是條件約束？」答對了！完全正確。

嚴格來說，從 SQL Server **系統層級（資料庫理論）**的角度來看，PK、UK、FK、Check、Default **全部都是條件約束（Constraint）**：

|約束類型|英文名稱|作用|
|---|---|---|
|主鍵約束|PRIMARY KEY Constraint|唯一 + 非 NULL|
|唯一約束|UNIQUE Constraint|唯一（允許 NULL）|
|外部索引鍵約束|FOREIGN KEY Constraint|參考完整性|
|檢查約束|CHECK Constraint|自訂邏輯|
|預設值約束|DEFAULT Constraint|預設值|
|非空約束|NOT NULL Constraint|不允許 NULL|

你可以用這段 SQL 親自查證：

```sql
-- 列出某張表的所有約束（會看到 PK、UK、FK、Check 全部都在）
SELECT name, type_desc
FROM sys.objects
WHERE parent_object_id = OBJECT_ID('你的資料表名稱')
  AND type_desc LIKE '%CONSTRAINT%';
```

那為什麼 SSMS 介面要分這麼多節點？SSMS 的分類是**從使用者管理的便利性出發**，不是從系統理論出發。它的邏輯是：把**「鍵類型」的約束**（PK、UK、FK）放到「索引鍵」節點，因為概念上都跟「鍵」有關；把**「非鍵類型」的約束**（Check、Default）放到「條件約束」節點。

> 🎯 **實務小技巧**：如果有人問你「這張表有幾個條件約束？」，你可以這樣回答：「從 SSMS 介面的『條件約束』資料夾來看有 X 個，但從系統層級（`sys.objects`）來查，連同 PK、UK、FK 一起算，實際上有 Y 個。」這樣的回答會讓資深同事覺得你很有料 😎

### 也要注意 FK 的分類位置

你可能會問：「外部索引鍵（FK）不也是一種限制嗎？為什麼它被放在『索引鍵』節點，不在『條件約束』節點？」

答案是——**它其實兩個身份都有**。嚴格來說 FK 也是條件約束的一種，SSMS 把它歸類在「索引鍵」節點是因為它以「鍵」的形式存在。如果你用 T-SQL 去查 `sys.foreign_keys`，會看到它在系統層級確實也屬於條件約束家族。遇到分類疑惑時，不用糾結，理解它的本質就好。

### 什麼該放資料庫，什麼放程式？

這是個很實際的問題：不是所有檢查都要丟給資料庫當壞人，否則會增加 DB 伺服器的負擔。

- **應該放資料庫**：核心商業邏輯，例如「帳戶餘額不能小於 0」、「訂單金額必須大於 0」。
- **不太需要放資料庫**：輕量檢查（如「密碼長度必須大於 8 碼」），這種交給前端或後端程式處理就好。

## 五、索引（Indexes）——查詢的加速器

從這一節開始，我們從「限制」的世界切換到「加速」的世界。索引的存在純粹是為了讓查詢更快。

### 用生活比喻理解

想像一本 500 頁的書，你要找「索引鍵」這三個字出現在哪裡：

- **沒有索引**：一頁一頁翻，這叫**全表掃描（Table Scan）**。
- **有索引**：直接查書末附錄「索引鍵 → 第 87 頁、第 203 頁」，這叫**索引搜尋（Index Seek）**。

索引讓資料庫能「跳著找資料」，大幅減少需要掃描的資料量。

### 索引的兩大類型

理解這兩類，對你寫 .NET Core 或使用 Entity Framework 很有幫助。

**叢集索引（Clustered Index）** 決定資料實體儲存的順序，一張表**只能有一個**（就像書只能有一種頁面順序）。通常主鍵預設就是叢集索引。

**非叢集索引（Non-Clustered Index）** 不改變資料實體順序，只是另外建立一份「排好序的指標」，類似書後面的索引頁。一張表可以有**很多個**非叢集索引。

### 主鍵、唯一鍵與索引的關聯（再次呼應）

還記得前面說的嗎？建立主鍵時，SQL Server 會自動建立一個唯一索引。同樣地，你設的唯一鍵也會在「索引」節點看到一個對應的唯一索引。這就是為什麼同一個物件會出現在多個節點。

更重要的是，**UK 自動附送「非叢集唯一索引」，讓你免費獲得加速效果**。這裡有個新人很容易忽略的好東西：

> **當你建立一個 UK 時，SQL Server 會自動幫你建立一個「非叢集的唯一索引（Non-Clustered Unique Index）」。**

UK 的運作機制是這樣的：

```
你設定 UK_Employees_Email
         ↓
SQL Server 自動建立一個「非叢集唯一索引」
         ↓
這個索引同時有兩個作用：
  ① 檢查唯一性（限制） ← 你主觀設 UK 是為了這個
  ② 加速以 Email 查詢的速度（加速） ← 免費附送！
```

白話翻譯：你本來設 UK 只是為了「防止 Email 重複」，但 SQL Server 為了做這件事，必須建立一個 B-Tree 來存放所有 Email 值（這樣插入新值時才能快速檢查有沒有重複）。這棵 B-Tree 就是一個索引，所以當你寫：

```sql
SELECT * FROM Employees WHERE Email = 'test@example.com';
```

SQL Server 會說：「喔？我剛好有一棵存 Email 的 B-Tree，直接拿來用吧！」然後就跑 Index Seek，速度超快。

#### 學長提醒：不要在設了 UK 的欄位上再手動建索引

這個「免費附送的加速效果」在實務上非常有價值。很多新手在設了 UK 的欄位上**再手動建一個索引**，結果建出重複索引，浪費儲存空間又拖慢寫入。**UK 設完就夠了，不用再加索引。**

### PK、UK、一般索引的關係總整理

|物件|底層結構|唯一性|叢集？|允許 NULL|
|---|---|---|---|---|
|**PK**|借用唯一索引|強制|預設叢集|❌|
|**UK**|借用唯一索引|強制|預設非叢集|✅（但只能一個）|
|**一般索引（Non-Clustered Index）**|自己就是索引|可選（通常不限制）|非叢集|✅（任意多個）|

### ⚠️ 踩坑警告：索引建太多反而拖垮系統

索引是用「**寫入時間**」和「**儲存空間**」來換取「**查詢速度**」。有些新手發現查詢很慢，就把所有欄位都加上索引。結果查詢變快了，**但每次有新使用者註冊（INSERT 資料）時系統卡得要命**——因為寫入一筆資料，資料庫要在背後同時更新好幾個索引目錄！

- **適合建索引的欄位**：經常出現在 `WHERE` 條件裡的欄位（例如 `OrderDate`、`CustomerId`）。
- **不要每個欄位都建索引**，要抓平衡。

## 六、統計資料（Statistics）——資料庫大腦的導航路況

這是最容易被忽略、但其實每天都在默默影響系統效能的東西。

### 它是什麼？

統計資料是 SQL Server 自己維護的一份「**資料分布摘要**」，例如某個欄位有多少不同的值、哪些值比較常出現、資料分佈是均勻還是極度傾斜。

### 它在解決什麼問題？

當你下一個 SQL 查詢時，SQL Server 的**查詢最佳化工具（Query Optimizer）**要決定：該用哪個索引？該用哪種 Join 方式？要從哪張表先開始查？

這些決策不是憑空做的，**而是根據統計資料來估算每種執行計畫的成本**。

### 用生活比喻理解

統計資料就像 **Google Maps 的路況資訊**。你查路線時，導航有時叫你走高速公路、有時叫你走平面道路，為什麼？因為它看得到哪裡在塞車。

SQL Server 也一樣：它會收集每個欄位資料的「分佈狀況」。例如它發現「性別」這個欄位 99% 都是男生，那你搜尋「男生」的時候，它就會判斷「與其查目錄（索引），不如直接整本書快速翻一遍還比較快」。

### ⚠️ 踩坑警告：莫名其妙變慢的兇手

實務上你很可能遇到這種情境：

> Portal 系統某張大表加了一堆新資料後，原本跑很快的查詢突然慢到超時。

很多時候原因不是 SQL 寫錯，也不是索引沒建，而是**統計資料還停留在舊的分布狀況**——例如剛匯入了 100 萬筆新資料，但 SQL Server 大腦還以為資料庫只有 10 筆，於是就會選錯找資料的策略。

這時手動執行 `UPDATE STATISTICS` 通常就能讓它重新變快。

### 為什麼節點裡有一堆 `_WA_Sys_...` 的東西？

SQL Server 預設會自動建立和更新統計資料（`AUTO_CREATE_STATISTICS` 和 `AUTO_UPDATE_STATISTICS` 選項），那些 `_WA_Sys_...` 開頭的項目，就是系統自動建立的。

另外，**每次你建立索引時，SQL Server 也會自動建立一份對應的統計資料**。這就是為什麼統計資料和索引的數量常常對得上。

## 七、整合理解：為什麼同一個東西會出現在不同節點？

這是這篇文章最關鍵的觀念，我用一個情境帶你串起來。

> **假設你在某張表建立了一個主鍵 `PK_Employees` 在 `EmployeeId` 欄位上。**

接下來會發生什麼？

|出現位置|為什麼會出現|
|---|---|
|✅ 索引鍵節點|因為它是個「鍵」|
|✅ 索引節點|因為它同時是個叢集索引|
|✅ 統計資料節點|因為索引建立時自動產生了統計資料|
|❌ 條件約束節點|不會出現，那裡只放 Check 和 Default|

這種「同一個東西多處出現」的設計，對初學者確實容易混淆，但理解後你會發現挺方便——**你可以從不同的管理角度切入同一個物件**。

## 八、深入辨析：唯一索引 vs 唯一鍵 vs 主鍵

這一節是進階內容，但非常重要——理解這三者的差異，你對資料庫的認識會提升一個層次。

### 一句話先記住結論

> **唯一索引是「工具」，PK 和 UK 是「規則」。規則「使用」工具來運作。**

### 三者的關係

```
     你建立 PK            你建立 UK         你建立一般唯一索引
         │                    │                    │
         ↓                    ↓                    ↓
 SQL Server 附送：    SQL Server 附送：        你直接建立：
   「唯一索引」          「唯一索引」            「唯一索引」
    (叢集)               (非叢集)               (非叢集)
         │                    │                    │
         └──────────┬──────────┴──────────┬─────────┘
                    ↓
          底層都是同一種東西：B-Tree 結構 + 唯一性檢查
```

### 用 C# 幫你秒懂

這就像在問：`public int Id { get; set; }` 和 `private int _id;` 是不是不同東西？

答案是：**是不同抽象層級的東西**。屬性是「對外的邏輯規則」，欄位是「底層的儲存機制」。屬性「使用」欄位來實作。

|層級|是什麼|例子|
|---|---|---|
|**邏輯層（規則）**|你告訴 SQL Server「我希望...」|PK、UK|
|**實體層（工具）**|SQL Server 實際拿來執行規則的東西|唯一索引|

**所以唯一索引「不等於」UK，但 UK「使用」唯一索引來實作自己。**

### 為什麼 NULL 的行為不一樣？

你可能會問：「既然 PK 和 UK 都用同一種唯一索引實作，為什麼 PK 不能 NULL 而 UK 可以？」

關鍵在這裡——**NULL 的規則不是由「唯一索引」決定的，而是由「它上面的邏輯規則（PK 或 UK）」決定的**。

|層級|規則對 NULL 的處理|
|---|---|
|**主鍵（PK）**|額外強制：NOT NULL（這是 PK 邏輯層加上去的規則）|
|**唯一鍵（UK）**|允許 NULL|
|**底下的唯一索引**|本身只關心「key 不能重複」，不管 NULL|

用 C# 類比就更清楚了：

```csharp
// 主鍵 = HashSet + 「不允許 null」的前置檢查
public void AddPrimaryKey(T value) {
    if (value == null) throw new Exception("PK 不能為 null");
    _hashSet.Add(value);  // 借用 HashSet 的去重機制
}

// 唯一鍵 = HashSet + 「允許 null」（但只能一個）
public void AddUniqueKey(T value) {
    _hashSet.Add(value);  // 一樣借用 HashSet 的去重機制
}
```

底層用同一個機制（唯一索引 / HashSet），但上面的規則決定要不要擋 NULL。

### 兩種建立方式的差別

你可以**單獨建立一個唯一索引**，而不建立 UK：

```sql
-- 方式 1：建立一個 UK（同時建立唯一索引 + 記錄一個「UK 約束」）
ALTER TABLE Employees ADD CONSTRAINT UQ_Email UNIQUE (Email);

-- 方式 2：只建立一個唯一索引（沒有 UK 約束的概念）
CREATE UNIQUE INDEX IX_Email ON Employees(Email);
```

兩種寫法執行後的**效果幾乎一樣**（Email 都不能重複），但在 SSMS 的呈現上有差別：

|建立方式|「索引鍵」節點|「索引」節點|系統層級條件約束|
|---|---|---|---|
|建立 UK（方式 1）|✅ 有|✅ 有|✅ 有（登記為 UNIQUE_CONSTRAINT）|
|建立唯一索引（方式 2）|❌ 沒有|✅ 有|❌ 沒有|

**實務建議**：如果你的目的是強調「這是一條業務規則」（例如 Email 是重要的業務欄位），建議用**方式 1**。語意清楚，未來維護好讀。如果只是純粹為了加速查詢 + 順便確保唯一，才考慮方式 2。大多數情況推薦用 UK。

## 九、五大節點總整理表

幫你把整篇收斂成一張表，未來忘記時翻這張就夠了：

|節點|生活比喻|實務任務|屬於「限制」還是「加速」|設計注意事項|
|---|---|---|---|---|
|**主鍵 (Primary Key)**|身分證字號|確保每筆紀錄唯一|限制（順帶建立索引）|用無意義流水號，不要用會變的資料|
|**索引鍵 (Keys)**|戶口名簿配偶欄|包含 PK / UK / FK，建立鍵類型規則|限制（FK 兼具關聯功能）|確保 FK 對應的主鍵真的存在|
|**條件約束 (Constraints)**|夜店保鑣|擋下不合規定的髒資料|限制|核心邏輯放這裡，輕量檢查放程式|
|**索引 (Indexes)**|字典的部首目錄|加速 SELECT 查詢|加速|常搜尋的欄位才建，建太多會拖慢 INSERT|
|**統計資料 (Statistics)**|導航路況情報|讓資料庫選出最快查詢路線|加速（輔助最佳化工具）|大量變更資料後，效能變差時記得更新它|

## 十、小思考練習 🤔

在進到下一篇實作篇之前，先用你工作上熟悉的場景動動腦：

> 假設你有一張 Portal 系統的訂單表 `Orders`，你會怎麼規劃這五個節點的內容？

試著對應看看：

- `OrderId` 該設成什麼？
- `OrderNumber`（對客戶顯示的訂單編號，也要不重複）該是什麼？
- `CustomerId`（指向客戶表）該是什麼？
- `OrderAmount > 0` 該是什麼？
- `CreatedDate DEFAULT GETDATE()` 又該是什麼？
- 如果經常用 `OrderDate` 來查詢報表，你會再加什麼？

把這個心智模型建立起來後，不管是看 SSMS 介面、寫 Entity Framework 的 Migration、還是看別人的資料庫 Schema，你都能很快理解設計者的意圖。
