# Part 1：JavaScript ES Modules 完整指南

> **閱讀順序**：這是第一篇，請先讀完這篇，再閱讀 Part 2。
> 
> **適合對象**：有 C# 或後端開發背景、剛開始接觸現代 JavaScript 模組化的開發者。

---

## 第一章：為什麼需要模組化？

在開始學語法之前，我們先來搞清楚「為什麼要有這個東西」。這很重要，因為理解了動機，你就不會覺得語法是在硬背。

想像一下，你和三位同事一起開發一個大型網頁。你寫了一個變數叫 `count`，同事也寫了一個 `count`。如果大家都把程式碼塞在同一個空間，這兩個 `count` 就會互相覆寫——這個現象叫做**全域污染（Global Pollution）**，是早期前端開發最頭痛的問題之一，常常導致系統在莫名其妙的地方崩潰。

我們需要一種機制，讓你的 `count` 和同事的 `count` 井水不犯河水，同時又能在有需要的時候安全地共用工具。這就是**模組化（Modularization）**要解決的問題。

**ES Module（簡稱 ESM）** 是目前 JavaScript 官方標準的模組化機制。整套機制的核心只有兩個關鍵字：**`export`（匯出）** 和 **`import`（匯入）**。

> **歷史補充：IIFE 是怎麼回事？**
> 
> 你可能在舊程式碼或文章裡看過 IIFE（Immediately Invoked Function Expression，立即執行函式）這個術語。在 ES Module 誕生之前，JavaScript 沒有官方的模組功能，前輩們只好利用「函式執行完、裡面的變數就不會外洩」這個特性，把程式碼包在一個立刻執行的函式裡，來模擬模組的效果。那是一種「土炮」解法。現在我們有了正式的 ES Module，已經不需要再寫 IIFE 了。你維護老舊系統時可能會看到它，了解它的歷史背景就夠了。

## 第二章：核心概念——export 與 import

用一個生活比喻來建立直覺：把每一個 JavaScript 檔案（`.js`）想像成一家家獨立的**專賣店**。

- **`export`（匯出）📦**：就像是店家把商品擺到櫥窗，貼上標籤說「這個工具開放給外面的人帶走」。
- **`import`（匯入）🛒**：就像是你拿著購物車，跑到別人的店裡，指定要拿哪一件商品，帶回自己的店裡用。

預設情況下，店裡所有東西都鎖在裡面，外面的人看不到、也拿不到——除非你明確掛上 `export` 的標籤。

---

## 第三章：具名匯出與具名匯入（Named Export / Named Import）

這是最基本、最常用的方式。

### 3.1 具名匯出

你可以在宣告的時候直接加上 `export`，也可以先宣告、最後再一次性匯出——這兩種寫法效果完全一樣：

```javascript
// utils.js

// 寫法 A：宣告時直接 export
export function add(a, b) {
  return a + b;
}

export function multiply(a, b) {
  return a * b;
}

// 寫法 B：先宣告，最後一次性 export（適合一眼看清楚這個檔案對外開放了什麼）
function subtract(a, b) {
  return a - b;
}

const PI = 3.14159;

export { subtract, PI };
```

### 3.2 具名匯入

在另一個檔案裡，用大括號 `{}` 精準指定你要哪些東西：

```javascript
// main.js

import { add, multiply, PI } from './utils.js';

console.log(add(5, 3));      // 輸出: 8
console.log(multiply(4, 2)); // 輸出: 8
console.log(PI);             // 輸出: 3.14159
```

> ⚠️ **新手最常踩的坑：import 的 `{}` 裡面放什麼？**
> 
> `{}` 裡面放的是「對方 `export` 出來的變數名稱」，不是檔案名稱。舉例來說，如果你要用 `utils.js` 裡的 `EmailValidator`，你寫的是 `import { EmailValidator } from './utils.js'`——不是 `import { utils }`，更不是 `import { SharedUtils }`。

> ⚠️ **路徑一定要寫 `./`，不能省略！**
> 
> 在瀏覽器的原生環境中，如果你寫 `from 'utils.js'`（沒有 `./`），系統會以為你要找的是第三方套件（就像去大賣場找，而不是去隔壁鄰居的店找），最後就會報錯。一定要寫成 `from './utils.js'`。同樣地，**`.js` 副檔名也不能省略**，這和使用 Webpack / Vite 這類打包工具的環境不同，那些工具會在背後幫你擦屁股，但原生瀏覽器不會。

## 第四章：預設匯出與預設匯入（Default Export / Default Import）

### 4.1 兩種匯出方式的比較

為什麼要有兩種匯出方式？因為使用情境不同：

有時候一個檔案裡裝了幾十個小工具（就像五金行 🛠️），你需要告訴對方確切想要哪一樣。但有時候一個檔案從頭到尾就只做「一件事」（就像招牌便當店 🍱），你走進去甚至不用說名字，直接說「招牌一份」就好。

| 特性       | 具名匯出（Named）            | 預設匯出（Default）       |
| :------- | :--------------------- | :------------------ |
| **比喻**   | 五金行 🛠️（有很多商品）         | 招牌便當店 🍱（主打一樣東西）    |
| **語法**   | `export const A = ...` | `export default A`  |
| **數量限制** | 一個檔案可以有**無限多個**        | 一個檔案**只能有 1 個**     |
| **匯入方式** | 必須加 `{}`，名字必須一樣        | **不加** `{}`，可以自己取名字 |

### 4.2 預設匯出的寫法

```javascript
// MathHelpers.js

const calculateTax = (price) => price * 0.05;

// 告訴系統：這個檔案的「招牌」就是它
export default calculateTax;
```

### 4.3 預設匯入——可以隨意命名，這是它的超能力

```javascript
// 使用原本的名字也可以
import calculateTax from './MathHelpers.js';

// 想幫它取綽號也完全沒問題！
import getTax from './MathHelpers.js';
```

系統只認一件事：沒有 `{}` 的匯入，就是去拿那個檔案的「招牌」，至於你要叫它什麼名字，你說了算。

> ⚠️ **最常踩坑的錯誤：把預設匯出塞進大括號**
> 
> 如果對方用的是 `export default`，但你在 import 時加了大括號：`import { calculateTax } from './MathHelpers.js'`，系統會報錯，因為它會去找一個「叫做 calculateTax 的具名匯出」，但那家店根本沒有這道小菜。

## 第五章：混搭使用——同時有具名與預設

一個檔案可以同時擁有「1 個預設匯出（招牌）」與「多個具名匯出（小菜）」。這在業界其實很常見，React 框架就是這樣設計的：React 核心是預設匯出，而 `useState`、`useEffect` 等工具是具名匯出。

```javascript
// utils.js

// 小菜（具名匯出）
export const add = (a, b) => a + b;
export const minus = (a, b) => a - b;

// 招牌（預設匯出）
const calculateTax = (price) => price * 0.05;
export default calculateTax;
```

匯入時，預設匯出寫在前面、不加大括號，具名匯出加大括號跟在後面，中間用逗號隔開：

```javascript
// 招牌放前面，小菜加大括號放後面
import getTax, { add, minus } from './utils.js';
```

> ⚠️ **最常犯的兩個錯誤**
> 
> 第一：忘記加逗號，寫成 `import getTax { add }`——這會直接報語法錯誤。
> 
> 第二：把預設匯出誤塞進 `{}`，寫成 `import { getTax, add }`——這樣系統會以為你要找名叫 `getTax` 的具名匯出，當然找不到。

## 第六章：進階技巧

### 6.1 重命名——`as` 關鍵字

**實務上解決的問題**：你的專案裡可能同時引入了 `dateUtils.js` 和 `stringUtils.js`，這兩個檔案都恰好有一個叫 `format` 的具名匯出。如果同時 import 進來，就會撞名。解法是用 `as` 幫其中一個取綽號：

```javascript
// 匯入時重命名，解決撞名問題
import { format as dateFormat } from './dateUtils.js';
import { format as stringFormat } from './stringUtils.js';

// 現在兩個都能安全使用了
dateFormat('2024-01-01');
stringFormat('hello world');
```

也可以在匯出時就先重命名：

```javascript
// 匯出時重命名
export { subtract as minus };
```

### 6.2 統包匯入——`* as`

**實務上解決的問題**：當一個檔案裡有幾十個工具時，在 `{}` 裡一個一個列出來很冗長。用 `* as 變數名` 可以把所有具名匯出打包成一個「工具箱物件」，要用哪個再用小數點取出：

```javascript
import * as Utils from './utils.js';

// 用小數點取工具
console.log(Utils.add(5, 3));   // 輸出: 8
console.log(Utils.PI);           // 輸出: 3.14159
console.log(Utils.subtract(10, 4)); // 輸出: 6
```

就像是去五金行跟老闆說「給我一整盒綜合工具箱 🧰」，而不是逐一點名。

### 6.3 重新匯出——Re-export（中繼站模式）

**實務上解決的問題**：隨著專案越來越大，`utils` 資料夾可能有 10 個不同的檔案。如果 `main.js` 需要用到這些工具，光是 import 就要寫滿 10 行，版面雜亂，而且每次都要記得哪個工具放在哪個檔案。

解法是建立一個**「中繼站」**（也稱為 Barrel Pattern 木桶模式）。這個中繼站自己不寫任何邏輯，只負責把各個檔案的工具彙整後統一對外提供：

```javascript
// utils/index.js（綜合大賣場，自己不生產，只彙整再出貨）

// 傳統兩步走：先買進來，再賣出去
import { add } from './math.js';
export { add };

// 縮寫寫法（Re-export）：直接轉交，效果完全一樣
export { add } from './math.js';
export { getToday } from './date.js';
```

`main.js` 現在只需要對準中繼站，一次買齊：

```javascript
// main.js：只需要 import 一個地方，乾淨俐落
import { add, getToday } from './utils/index.js';
```

> ⚠️ **重要陷阱：Re-export 中的工具，在那個檔案內部是「過客」，不是「住客」**
> 
> 如果你在 `index.js` 寫了 `export { add } from './math.js'`，這個 `add` 是被「直接穿透轉交」的，並沒有被匯入到 `index.js` 的作用域裡。所以你不能在 `index.js` 的其他地方直接呼叫 `add()`——系統會說 `add is not defined`。
> 
> 如果你在這個中繼站「既要轉交給別人、自己又要用」，就必須乖乖拆成兩行：先 `import` 進來，再 `export` 出去。

### 6.3.1 實戰情境：用 Re-export 解決多層依賴鏈

回到第七章會提到的這個專案結構：

```
index.html → main.js → utils.js → numberValidator.js
```

`index.html` 如果想用 `numberValidator.js` 裡的工具，預設必須自己寫一行很長的路徑 import，而且還要知道那個檔案藏在哪個資料夾。這很不方便，也讓使用端暴露在底層的檔案結構細節裡。

用 Re-export 可以讓每一層中繼站負責往上傳遞，讓 `index.html` 只需要認識 `main.js` 這一個入口：

**第一層：最底層工具（不動）**

```javascript
// numberValidator.js
export const isPositive = (number) => number > 0;
export const isInteger = (number) => Number.isInteger(number);
```

**第二層：utils.js——自己用 + 順手往上轉交**

注意這裡有兩行分開的語法，用途不同：

```javascript
// utils.js
import { isPositive } from './numberValidator.js'; // 自己要用來寫 checkAge，只需要 isPositive

export function checkAge(age) {
    if (isPositive(age)) {
        console.log(`年齡 ${age} 驗證通過`);
    }
}

// 轉交給上層使用——兩個工具都要往上送
export { isPositive, isInteger } from './numberValidator.js';
```

> 💡 **為什麼 `import` 那行只寫 `isPositive`，`export` 那行卻要寫兩個？**
> 
> 這兩行的目的完全不同。`import { isPositive }` 是「`utils.js` 自己要用」，`checkAge()` 裡面只呼叫了 `isPositive`，所以只拿這一個就夠了。`export { isPositive, isInteger }` 是「幫上層轉交」，既然 `numberValidator.js` 提供了兩個工具，就應該都往上送，讓上層可以自由選用。

**第三層：main.js——統一對外的入口**

```javascript
// main.js
export { checkAge } from './utils.js';
export { isPositive, isInteger } from './utils.js'; // 繼續往上轉交
```

**最終：index.html 只需要認識 main.js**

```html
<script type="module">
    import { isPositive, isInteger, checkAge } from './main.js';

    console.log(isPositive(5));  // ✅ true
    console.log(isInteger(3.5)); // ✅ false
    checkAge(25);                // ✅ 年齡 25 驗證通過
</script>
```

這樣做最大的好處是：以後就算你把 `numberValidator.js` 搬到別的資料夾、或是拆成更多檔案，**只需要修改中間層的 re-export 路徑**，`index.html` 完全不用動，因為它只認識 `main.js` 這個入口。

### 6.4 統包重新匯出——`export * from`

如果你想把某個檔案的所有具名匯出全部轉交出去，可以用 `*`：

```javascript
// index.js
// 把 math.js 裡所有的具名匯出，全部對外上架
export * from './math.js';
```

> ⚠️ **進階陷阱：`export *` 不包含預設匯出！**
> 
> `export * from './math.js'` 只會轉交所有的 <mark style="background: #FF5582A6;">具名</mark> 匯出（小菜），但**不會自動轉交預設匯出（招牌）**。如果你也想轉交招牌，必須額外多寫一行：
> 
> ```javascript
> export * from './math.js';              // 轉交所有具名
> export { default } from './math.js';   // 額外指定轉交招牌
> ```

### 6.5 揭開 default 的底層秘密

這裡有一個很多人不知道的底層知識，搞懂它之後，你對整套 ES Module 的理解會更紮實。

**「預設匯出」在系統底層，其實只是一個名字剛好叫做 `default` 的具名匯出。**

當你寫下 `export default myTool` 時，系統底層實際上是：

```javascript
// 你寫的
export default myTool;

// 系統底層真實的樣子
export { myTool as default };
```

系統把你的 `myTool` 放進了一個標籤叫做 `default` 的專屬櫃子。這解釋了兩件看起來很神奇的事：

**為什麼 import 招牌可以隨便取名？** 因為系統只是去找標籤叫 `default` 的櫃子，拿出來後交給你自己取的任何名字。

**為什麼大賣場轉交招牌時要寫 `export { default }`？** 因為大賣場要用「標籤名稱」來叫貨，而招牌的標籤名就是 `default`。

所以，我們平常習慣寫的：

```javascript
import myTool from './math.js';
```

其實就是下面這行的**語法糖（快速縮寫）**：

```javascript
import { default as myTool } from './math.js';
```

兩種寫法效果完全一樣，前者只是系統給你的貼心捷徑。

## 第七章：在網頁中串接運行

### 7.1 關鍵屬性：`type="module"`

在 HTML 裡要讓瀏覽器看懂 ES Module 的 `import / export`，你必須在 `<script>` 標籤加上 `type="module"`，明確告訴瀏覽器：「這不是普通的腳本，這是一個模組！」

```html
<!DOCTYPE html>
<html>
<head>
    <title>模組化測試</title>
</head>
<body>
    <h1>請打開開發者工具 (F12) 的 Console 看結果</h1>

    <!-- 只需要載入最頂層的主程式，其餘交給 import 自動串接 -->
    <script type="module" src="./main.js"></script>

    <!-- 也可以直接在 HTML 裡寫模組程式碼 -->
    <script type="module">
        import { add } from './utils.js';
        console.log(add(1, 2));
    </script>
</body>
</html>
```

### 7.2 依賴圖——瀏覽器如何「順藤摸瓜」

一個很重要的觀念：你的 HTML 只需要載入最頂層的主程式（`main.js`），**不需要**把每一個 `.js` 檔案都一一列在 HTML 裡。瀏覽器會順著 `import` 的線索，自動把所有依賴的檔案抓回來。

假設專案結構是 `index.html → main.js → utils.js → numberValidator.js`，瀏覽器的內心戲是這樣的：

1. 看到 HTML 說要載入 `main.js`，去下載。
2. 打開 `main.js` 一看有 `import './utils.js'`，先暫停，跑去下載 `utils.js`。
3. 打開 `utils.js` 一看有 `import './numberValidator.js'`，再暫停，跑去下載。
4. `numberValidator.js` 沒有依賴別人了。這時才開始**由下往上**真正執行程式碼。

這個像樹枝展開的結構，在實務上叫做**「依賴圖（Dependency Graph）」**。以後你的 `utils.js` 再多引入幾個底層工具，都不需要去改 `index.html`，因為一切都在 JavaScript 內部的 `import` 鏈裡自動串好了。

### 7.3 重要觀念：「瀏覽器抓到」≠「你可以直接用」

這裡有一個很容易讓人誤解的地方，一定要說清楚。

在上面的例子裡，`index.html` 載入 `main.js`，瀏覽器順藤摸瓜把 `numberValidator.js` 也抓回來了——**但這不代表 `index.html` 就能直接使用 `numberValidator.js` 裡的函式**。

原因在於：**每個模組的作用域是完全獨立的**。`main.js` import 了 `numberValidator`，只代表 `main.js` 自己可以用，這個存取權限不會「外洩」給 `index.html` 或其他任何人。

如果你想在 `index.html` 的 `<script type="module">` 裡直接呼叫 `numberValidator` 的函式，**你必須自己明確寫 import**：

```html
<script type="module">
    <!-- ❌ 錯誤：以為 main.js 幫你帶進來了 -->
    <!-- isPositive(5); -->  <!-- 報錯：isPositive is not defined -->

    <!-- ✅ 正確：要用就要自己 import -->
    import { isPositive } from './numberValidator.js';
    console.log(isPositive(5)); // 這樣才能用
</script>
```

一句話記住：**每個檔案要用什麼，就要自己去 import，沒有人能幫你代領。**

### 7.4 常見踩坑：CORS 錯誤

當你寫好程式，直接用滑鼠點兩下打開 `index.html`（網址列顯示 `file:///C:/...`）時，通常什麼都不會發生。按下 F12 會看到紅字，通常包含 **CORS policy** 或 **Cross-Origin** 等字眼。

原因是：基於安全性考量，瀏覽器不允許網頁透過 `file://` 通訊協定去硬碟裡 `import` 其他檔案。你必須啟動一個**本地端伺服器（Local Server）**。實務上最簡單的做法，就是在 VS Code 安裝 **「Live Server」** 擴充套件，點擊右下角的 "Go Live" 按鈕，網址列就會變成 `http://127.0.0.1:5500/...`，問題就解決了。

## 第八章：模組作用域——模組內 vs 模組外

### 8.1 傳統腳本 vs 模組：一道防護網

|比較項目|傳統 `<script src="...">`|模組化 `<script type="module">`|
|:--|:--|:--|
|**預設狀態**|門戶洞開的大廣場（全域作用域）|門禁森嚴的大樓（模組作用域）|
|**變數可見度**|宣告的變數，所有檔案都看得到|沒寫 `export`，外面絕對看不到|
|**安全性**|極低，容易互相覆蓋|極高，檔案之間的變數互不干擾|

在模組的世界裡，每一個 `.js` 檔案都像是一棟有嚴格門禁的辦公大樓。你在大樓裡宣告的任何變數、函式，預設只有這棟樓裡的程式碼看得到。`export` 就是那個「對外服務窗口」——唯一的合法對外管道。

```javascript
// userManager.js

// 大樓內部的機密，沒有 export（外面的人絕對看不到）
const adminPassword = "super_secret_123";

// 內部的輔助函式，不對外開放
const logActivity = () => {
    console.log("有會員活動了");
};

// 對外開放登入功能——外部可以呼叫這個方法，但看不到 adminPassword
export const login = (pwd) => {
    if (pwd === adminPassword) {
        console.log("登入成功！");
    }
};
```

如果你在 `main.js` 試圖直接讀取 `adminPassword`，系統會回報 `ReferenceError: adminPassword is not defined`——就算你打開瀏覽器的 F12 Console 手動輸入也一樣，因為你站的位置是「全域大馬路」，根本走不進那棟大樓。

> 💡 **實務原則：能不 `export` 就不要 `export`，保持模組的乾淨。** 只開放別人明確會用到的東西，底層邏輯、密鑰、輔助工具都應該藏起來。

### 8.2 匯出的變數是「唯讀的」——Live Bindings

這是一個非常經典的進階陷阱：當你把變數 `export` 出去，別人 `import` 進來後，**這個變數對買家來說是唯讀（Read-only）的**。

```javascript
// userManager.js
export let activeUsersCount = 10;
export const addUser = () => { activeUsersCount++; }; // 只有原廠能改
```

```javascript
// main.js
import { activeUsersCount } from './userManager.js';

// 你可以讀取它
console.log(activeUsersCount); // 10

// ❌ 但你不能直接改它！
activeUsersCount = 11; // 報錯：Assignment to constant variable
```

如果真的需要修改，必須請原廠提供一個修改的方法（如上面的 `addUser()`），讓你呼叫那個方法來間接修改。這個設計讓資料的更動路徑清晰且可控。

## 第九章：模組快取與重要注意事項

### 9.1 模組快取（Module Caching）——每個模組只執行一次

如果 `main.js` 和 `utils.js` 都 `import` 了同一個 `numberValidator.js`，瀏覽器只會下載並執行它一次。之後所有的 `import` 都只是拿到已經執行過的結果的「參照」。

這個機制叫做**「模組快取（Module Caching）」**，有兩個好處：一是節省效能與網路資源；二是確保「狀態共享」——如果 `numberValidator.js` 裡有一個 `let count = 0`，所有引入它的檔案都共用同一個 `count`，不會因為多次 import 而重複歸零。

### 9.2 四條重要注意事項

**第一：`import` 只能寫在最頂層。** `import` 不能寫在 `if` 條件判斷或函式裡面。原因是瀏覽器在真正執行程式碼之前，會先快速掃描所有的 `import`，建立完整的依賴圖，然後才開始下載和執行。這個機制稱為**「靜態分析（Static Analysis）」**。如果 `import` 藏在 `if` 裡，瀏覽器執行前無法確定到底需不需要這個檔案，系統就無法正常運作。

**第二：ES Modules 始終在嚴格模式（`'use strict'`）下運行。** 不需要你手動宣告，這是模組的預設行為。

**第三：在瀏覽器環境中，路徑必須完整，包含副檔名。** 寫 `'./utils.js'`，而不是 `'./utils'`。

**第四：ES Modules 會被延遲執行。** 效果相當於加上 `defer` 屬性，所以 `<script type="module">` 不會阻塞 HTML 的解析，網頁會先把 HTML 架構建好，再來執行模組裡的程式碼。

## 章節總結

讀完這一篇，你應該對以下概念有了清楚的掌握：

**模組化的動機**是解決全域變數衝突，讓程式碼可以分檔管理。**`export`** 是把工具上架，**`import`** 是把工具買回來用。

匯出有兩種方式：**具名匯出**像五金行，用 `{}` 精準買工具，名字要和對方一樣；**預設匯出**像招牌便當店，不加 `{}`，買回來可以自己取名字。一個檔案可以同時擁有兩種。

進階技巧方面，**`as`** 用來重命名、解決撞名；**`* as`** 用來統包成一個物件；**Re-export** 用來建立中繼站、整理龐大的工具庫。

在 HTML 串接上，用 `<script type="module" src="...">` 載入主程式，瀏覽器會自動順藤摸瓜。記得用 Local Server，不要直接雙擊 HTML。
