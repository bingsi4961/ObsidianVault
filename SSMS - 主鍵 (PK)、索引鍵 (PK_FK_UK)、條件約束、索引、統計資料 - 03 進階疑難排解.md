## Q1：「索引/索引鍵」視窗是什麼？Unique Key 又在哪裡設定？

很多人直覺以為「唯一性 = 一種規則限制 = 應該在條件約束裡」，翻了半天找不到，**其實它藏在「索引/索引鍵」視窗裡**。

原因是：主鍵、唯一鍵、一般索引，底層全部都是用 **B-Tree** 結構實作的。SQL Server 的設計原則是「底層機制相同的東西，統一用一個介面管理」，所以 SSMS 把這三者都集中在「索引/索引鍵」這個中央控制台，而不是放到「條件約束」。

設定步驟：在資料表 Design 畫面空白處右鍵 → **「索引/索引鍵 (Indexes/Keys)」** → 點擊「加入 (Add)」 → 在右側屬性欄位中，將「**類型 (Type)**」從「索引」改為「**唯一鍵 (Unique Key)**」 → 選擇欄位並給一個有意義的名稱（例如 `UQ_Employees_Email`）。

這個設計也解釋了主鍵為何會同時出現在以下位置：

- **索引鍵** 節點
- **索引** 資料夾
- **索引/索引鍵** 視窗
- **統計資料** 資料夾

## Q2：為什麼「條件約束」資料夾有東西，但「檢查條件約束」對話框是空的？

你描述的情境是：資料表的「**條件約束**」資料夾下已經有好幾個 default constraints，但你在 Design 畫面右鍵點「**檢查條件約束**」打開後，裡面竟然是空的。為什麼？

答案是：它們是不同性質的約束，顯示位置不同。SQL Server 把「條件約束」分成不同類型：

|類型|性質|顯示位置|
|---|---|---|
|**檢查約束 (Check Constraint)**|邏輯判斷，例如 `Age > 18`|「**檢查條件約束**」對話框|
|**預設值約束 (Default Constraint)**|填補機制，沒給值就代入預設值|「設計」畫面下方的「**資料行屬性**」，或「**條件約束**」資料夾|

所以你看到的情況是完全正常的：「條件約束」**資料夾**會把 default 和 check **全部**列出來；但「**檢查條件約束**」對話框只會列出 **check 類型**的項目，default 不會在這裡出現。

如果你想看 default 的設定，請去看「資料行屬性」中的「**預設值或繫結**」欄位。

## Q3：修改 FK 時為什麼跑出兩個項目？我明明只點了一個 FK！

你的情境是：「索引鍵」資料夾裡已存在一個 PK、一個 FK，你在那唯一的 FK 上按右鍵選「修改」，結果跳出的視窗卻顯示兩個 FK，這是怎麼回事？

### 答案：關聯性視窗會連「別的表指向我的 FK」一起顯示

當你對 FK 按右鍵選「修改」，跳出的對話框其實是「**外部索引鍵關聯性 (Foreign Key Relationships)**」管理員。這個視窗的設計是：**列出所有跟這張表有關的關聯**，包含兩個方向：

- **出向（outgoing）**：這張表自己擁有的 FK（這張表是子表，參考別人）
- **入向（incoming）**：別的表指向這張表的 FK（這張表是父表，被別人參考）

所以你會看到比「索引鍵」節點更多的項目，這**完全是正常行為**，不是 SSMS 出錯，也不是歷史殘留。

把這兩個視窗的顯示範圍整理清楚：

|項目|「索引鍵」節點（物件總管）|「關聯性」視窗（Design → 右鍵）|
|---|---|---|
|顯示「自己的 FK（出向）」|✅ 會|✅ 會|
|顯示「別人指向我的 FK（入向）」|❌ **不會**|✅ **會顯示**|

為什麼有這個差別？因為「索引鍵」節點的邏輯是「**有哪些鍵類型物件登記在我名下？**」——FK 在系統層級是登記在「子表」名下的，所以父表看不到入向 FK。而「關聯性」視窗的邏輯是「**我（這張表）跟其他表之間存在哪些關聯？**」——不論出向還是入向，都算是「我參與的關聯」。

### 修改時怎麼判斷你在改哪個 FK？

既然這個視窗會同時顯示多方向的 FK，修改時務必注意：看**左側清單的關聯名稱**（FK 命名通常是 `FK_子表_父表` 的格式，從名稱就能判斷方向），再看**右側「資料表及資料行規格」**，它會明確寫出「主索引鍵資料表」和「外部索引鍵資料表」是誰。確認清楚之後再動手。
